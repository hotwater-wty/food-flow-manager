# V2 Redis 应用场景规划

本文档用于整理 V2 阶段 Redis 的落地边界和候选场景，辅助判断哪些功能适合当前 V2 完成，哪些更适合后续 V3 或扩展阶段。

当前项目已经完成用户端启售菜品列表缓存，并在菜品新增、修改、删除、状态变更时清除相关缓存。后续 Redis 的使用应继续遵循一个原则：

```text
每个 Redis 技术点必须对应明确业务场景，不为了堆技术而引入复杂组件。
```

## 1. 当前 Redis 使用现状

### 1.1 已完成场景

用户端启售菜品列表缓存：

```text
GET /api/user/dishes
GET /api/user/dishes?categoryId=1
```

当前实现要点：

- 使用 `StringRedisTemplate + ObjectMapper` 缓存 `List<DishVO>`。
- Redis 中保存普通 JSON 字符串。
- 按全部启售菜品和分类启售菜品分别构建 key。
- 菜品新增、修改、删除、上下架后清除缓存。

推荐 key：

```text
foodflow:dish:on-sale:all
foodflow:dish:on-sale:category:{categoryId}
```

该场景已满足 V2 计划中“Redis 在真实业务场景中落地”的基本要求。

### 1.2 已沉淀的技术经验

当前已形成的 Redis 经验包括：

- 缓存读取、缓存未命中回源数据库、写入缓存的旁路缓存模式。
- 数据变更后删除缓存的缓存失效策略。
- `StringRedisTemplate + ObjectMapper + TypeReference` 处理泛型集合缓存。
- `GenericJackson2JsonRedisSerializer` 不适合作为复杂泛型集合缓存的首选方案。

对应专题文档：

```text
assets/过程资产/专题文档/Redis缓存JSON序列化与StringRedisTemplate方案.md
```

## 2. Redis 候选场景总览

| 场景 | 当前价值 | Redis 技术点 | 难度 | 建议阶段 |
|---|---:|---|---:|---|
| 启售菜品列表缓存 | 高，已完成 | String 缓存、JSON 序列化、缓存失效 | 中 | V2 已完成 |
| 账号状态缓存 | 高 | 缓存未命中回源、启禁用后同步更新或删除缓存、拦截器读缓存 | 中 | V2 推荐 |
| 用户端启用分类列表缓存 | 中 | 简单列表缓存、分类变更后失效 | 低 | V2 可选 |
| 用户端菜品详情缓存 | 中 | 单对象缓存、详情查询、停售校验 | 低-中 | V2 可选 |
| 防重复提交 Token | 中-高 | `SETNX`、短 TTL、幂等 Token | 中 | V2 可选强化 |
| 桌位状态或空闲桌位缓存 | 中但一致性风险高 | 高频失效、状态一致性 | 中-高 | 不建议 V2 优先 |
| 验证码登录 | 技术练习价值高，但属于新增登录链路 | 验证码 TTL、发送频率限制、一次性校验 | 中 | V3 或 V2 后半段 |
| 催单频率控制 | 中 | Redis 频率控制、业务冷却时间 | 中 | V3 |
| Token 黑名单或退出登录 | 中 | Token 黑名单、TTL 对齐 JWT | 中 | V3 |
| Redisson 分布式锁 | 技术复杂度高，当前必要性低 | 分布式锁、锁过期、看门狗 | 高 | V3，多实例实验时做 |
| 管理端分页结果缓存 | 低 | 分页 key 组合、复杂失效 | 中 | 不建议 |
| 当前开台会话缓存 | 低且风险高 | 强一致状态缓存 | 高 | 不建议 |

## 3. V2 推荐继续完成的 Redis 场景

### 3.1 账号状态缓存

推荐优先级：V2 推荐。

当前项目的拦截器会在每次请求中查询用户或员工账号状态，以保证账号被禁用后，旧 token 也不能继续访问系统。

当前问题：

```text
每次请求都查数据库，逻辑可靠但访问频率较高。
```

推荐使用 Redis 缓存账号状态：

```text
foodflow:account:user:status:{userId}
foodflow:account:employee:status:{employeeId}
```

推荐流程：

```text
拦截器读取账号状态
  -> 先查 Redis
  -> 未命中则查数据库
  -> 数据库查到后写入 Redis
  -> 判断账号是否正常
```

账号启用或禁用时：

```text
更新数据库
  -> 删除或更新对应账号状态缓存
```

关键原则：

```text
账号禁用必须尽快生效。
```

因此账号状态缓存不能只依赖较长 TTL 自然过期，启用、禁用员工或用户时必须同步处理缓存。

### 3.2 用户端启用分类列表缓存

推荐优先级：V2 可选。

用户端查询启用菜品分类列表也是读多写少场景，适合用 Redis 巩固列表缓存写法。

推荐 key：

```text
foodflow:dish-category:enabled:list
```

失效时机：

- 新增分类。
- 修改分类。
- 删除分类。
- 启用分类。
- 禁用分类。

该场景业务风险较低，适合作为菜品缓存之后的练习项。

### 3.3 防重复提交 Token

推荐优先级：V2 可选强化。

如果希望练习 Redis 的原子操作，可以在创建预约、创建订单等关键接口上增加防重复提交能力。

可使用 Redis 的 `SET NX EX` 思路：

```text
SET key value NX EX 30
```

含义：

- `NX`：key 不存在时才写入。
- `EX 30`：设置 30 秒过期时间。
- 写入成功表示第一次提交。
- 写入失败表示重复提交。

示例 key：

```text
foodflow:idempotent:{loginType}:{userId}:{requestToken}
```

该方案会影响接口调用方式，需要前端或 Apifox 携带请求幂等 token，因此建议放在账号状态缓存之后。

## 4. 不建议 V2 优先处理的场景

### 4.1 桌位状态或空闲桌位缓存

桌位状态会被多个主流程频繁修改：

- 创建预约。
- 取消预约。
- 预约扫码开台。
- 未预约扫码占座。
- 创建订单。
- 清台释放桌位。
- 启用或禁用桌位。

这些状态变化分散在多个模块中，缓存失效点多，且状态一致性要求高。当前 V2 已经重点处理了并发和数据库一致性，不建议马上把桌位状态放入缓存增加复杂度。

结论：

```text
桌位状态以数据库为准，V2 不优先缓存。
```

### 4.2 验证码登录

验证码登录很适合 Redis，但它不是单纯缓存增强，而是新增登录链路。

完整流程包括：

```text
发送验证码
  -> Redis 存储验证码并设置 TTL
  -> 限制验证码发送频率
  -> 校验验证码
  -> 注册或登录
  -> 删除验证码
```

该功能可以练习 TTL、频率控制、一次性校验，但会引入新的用户登录方式和接口设计，因此更适合 V3，或 V2 后半段在主任务完成后作为扩展。

### 4.3 Redisson 分布式锁

当前项目的关键并发问题主要通过以下方式处理：

- 数据库条件更新。
- 唯一约束。
- 事务。

这些方式已经适合当前单体项目和核心状态流转场景。Redisson 分布式锁不是更高级的必然替代品，而是在多实例部署、跨资源临界区、数据库约束不好表达时才更有必要。

结论：

```text
V2 不优先引入 Redisson。
```

可以在 V3 或多实例部署实验时再设计一个明确场景。

## 5. 推荐 V2 Redis 收束路线

当前 V2 推荐完成：

```text
1. 启售菜品列表缓存
2. 账号状态缓存
```

如果希望多练一个 Redis 场景，可以二选一：

```text
3A. 用户端启用分类列表缓存
3B. 防重复提交 Token
```

建议优先级：

```text
账号状态缓存 > 启用分类列表缓存 > 防重复提交 Token
```

原因：

- 账号状态缓存已经写入 V2 待办，且和当前鉴权逻辑直接相关。
- 启用分类列表缓存实现简单，能巩固当前已经验证通过的 JSON 缓存方案。
- 防重复提交 Token 工程价值更高，但会影响接口调用约定，适合作为 V2 强化项。

## 6. V3 或后续阶段候选

后续可以考虑：

- 手机验证码登录。
- 验证码发送频率控制。
- 催单频率控制。
- Token 黑名单和退出登录。
- Redisson 分布式锁。
- 多实例部署下的 Redis 锁实验。
- 更完整的缓存穿透、击穿、雪崩治理。
- 本地缓存 + Redis 的多级缓存。

这些内容更适合在 V2 主目标完成后继续扩展，避免当前阶段因为 Redis 练习过度而偏离 V2 的主线。

## 7. 当前结论

V2 阶段 Redis 不应只停留在“用过一次”，但也不需要全面铺开。

当前最合理的边界是：

```text
用菜品缓存证明 Redis 能服务高频查询；
用账号状态缓存证明 Redis 能服务鉴权链路；
可选用分类缓存或防重复提交补充练习；
把验证码、Redisson、复杂限流放到后续阶段。
```

这样既能形成清晰的技术亮点，也不会破坏当前 V2 的主线节奏。
