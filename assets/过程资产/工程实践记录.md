## 1. 用数据库唯一约束兜底业务唯一性

### 适用问题

当业务规则要求“某类数据绝对不能重复”时，不能只依赖 Java 代码中的查询判断，还应尽量让数据库通过唯一约束承担最终兜底。

典型例子：

- 用户手机号不能重复。
- 员工手机号不能重复。
- 预约编号、订单编号不能重复。
- 同一个用户同一时间只能有一个活跃开台会话。
- 同一张桌位同一时间只能绑定一个活跃开台会话。

### 当前项目问题

V2 并发控制中，预约扫码开台和非预约扫码占座都可能创建 `dining_session`。

业务上要求：

```text
一个用户同一时间最多只能有一个等待中或用餐中的开台会话。
```

V1/V2 初期代码可以先查询当前用户是否已有活跃会话：

```java
List<DiningSession> currentSessions = query()
        .eq("user_id", LoginContext.getUserId())
        .in("status", DiningSessionStatusEnum.WAITING,
             DiningSessionStatusEnum.DINING)
        .list();
```

这能处理普通请求，但不能彻底处理并发。

并发场景下可能出现：

```text
请求 A：查询用户是否有活跃会话 -> 没有
请求 B：查询用户是否有活跃会话 -> 没有

请求 A：插入 WAITING 会话
请求 B：插入 WAITING 会话

结果：同一个用户出现两条活跃会话
```

`@Transactional` 只能保证单个事务内部一致，不能自动阻止两个事务在插入前都读到相同旧数据。

### 核心思路

代码校验用于提前给出友好业务提示，数据库唯一约束用于并发场景最终兜底。

也就是说：

```text
Service 层查询校验：表达业务意图，提升提示友好度。
数据库唯一索引：保证底线规则不被并发击穿。
```

凡是“绝对不能重复”的规则，都应优先思考能否落到数据库唯一约束上。

### active_flag 的由来

`dining_session.status` 当前有四种状态：

```text
WAITING    等待中
DINING     用餐中
COMPLETED  已完成
CANCELED   已取消
```

业务上真正关心的是：

```text
WAITING / DINING = 活跃会话
COMPLETED / CANCELED = 历史会话
```

如果直接基于 `status` 建唯一约束，很难表达：

```text
同一个 user_id 下，status in (WAITING, DINING) 的记录最多只能有一条。
```

因此引入一个派生字段：

```text
active_flag
1    活跃
NULL 非活跃
```

创建等待中或用餐中的会话时，`active_flag = 1`。

会话完成或取消后，`active_flag = NULL`。

### 为什么非活跃状态用 NULL

如果使用：

```text
active_flag = 1 表示活跃
active_flag = 0 表示非活跃
```

再建立唯一索引：

```sql
UNIQUE KEY uk_user_active_session (user_id, active_flag)
```

就会产生问题：

```text
同一个用户只能有一条 active_flag = 0 的历史会话。
```

这显然不符合业务，因为用户可以有很多已完成或已取消的历史会话。

MySQL 的唯一索引允许多条记录在索引列中出现 `NULL`。因此采用：

```text
active_flag = 1    参与唯一约束，限制活跃会话只能有一条。
active_flag = NULL 不互相冲突，允许保留多条历史会话。
```

### 数据库设计参考

新增字段：

```sql
ALTER TABLE dining_session
ADD COLUMN active_flag TINYINT NULL COMMENT '活跃标识：1活跃，NULL非活跃';
```

初始化历史数据：

```sql
UPDATE dining_session
SET active_flag = CASE
    WHEN status IN (0, 1) THEN 1
    ELSE NULL
END;
```

增加用户活跃会话唯一约束：

```sql
ALTER TABLE dining_session
ADD UNIQUE KEY uk_dining_session_user_active (user_id, active_flag);
```

如需进一步约束同一桌位不能出现多个活跃会话，也可以增加：

```sql
ALTER TABLE dining_session
ADD UNIQUE KEY uk_dining_session_table_active (table_id, active_flag);
```

### 代码落点参考

实体增加字段：

```java
private Integer activeFlag;
```

创建开台会话时设置活跃：

```java
DiningSession diningSession = DiningSession.builder()
        .sessionNo(NumberUtils.generateSessionNo())
        .userId(LoginContext.getUserId())
        .tableId(tableId)
        .status(DiningSessionStatusEnum.WAITING)
        .activeFlag(1)
        .openTime(now)
        .createTime(now)
        .updateTime(now)
        .build();
```

取消或完成会话时释放活跃标识：

```java
session.setStatus(DiningSessionStatusEnum.COMPLETED);
session.setActiveFlag(null);
session.setCloseTime(now);
session.setUpdateTime(now);
```

Service 层仍可保留查询：

```java
boolean exists = lambdaQuery()
        .eq(DiningSession::getUserId, LoginContext.getUserId())
        .eq(DiningSession::getActiveFlag, 1)
        .exists();

if (exists) {
    throw new BusinessException("当前用户已存在用餐会话");
}
```

但要明确：这只是友好提示，不是并发最终防线。

### 唯一索引冲突处理

并发插入时，数据库可能抛出唯一索引冲突异常。

推荐做法：

- 全局异常处理器兜底处理 `DuplicateKeyException`。
- 对需要更精确提示的业务方法，可以局部捕获并转换成 `BusinessException`。

示例：

```java
try {
    save(diningSession);
} catch (DuplicateKeyException e) {
    throw new BusinessException("当前用户已存在用餐会话，请勿重复开台");
}
```

### 适用原则

遇到类似问题时，可以按这个顺序思考：

```text
1. 这条业务规则是不是绝对不能被突破？
2. 如果两个请求并发进入，单纯先查再写是否会失效？
3. 能否把规则转成唯一索引、条件更新、外键、非空约束等数据库约束？
4. Java 代码是否只负责友好提示和业务流程，数据库负责最终底线？
5. 唯一约束冲突后，是否有明确的业务异常提示？
```

### 当前项目后续落点

V2 处理开台并发时，建议按以下顺序推进：

1. `dining_session` 增加 `active_flag` 字段。
2. 创建预约开台、非预约开台会话时写入 `active_flag = 1`。
3. 取消等待中会话、清台完成会话时写入 `active_flag = NULL`。
4. 增加 `(user_id, active_flag)` 唯一索引。
5. 视情况增加 `(table_id, active_flag)` 唯一索引。
6. 保留 Service 层查询校验，用于提前提示。
7. 对唯一索引冲突做业务异常转换。

### 面试表达素材

在处理开台并发时，我没有只依赖 Service 层的“先查再插”判断，因为两个请求并发进入时可能都读到没有活跃会话。我的做法是把“同一用户只能有一个活跃会话”抽象成数据库约束：增加 `active_flag` 字段，活跃会话为 `1`，历史会话为 `NULL`，再建立 `(user_id, active_flag)` 唯一索引。这样 Service 层校验负责友好提示，数据库唯一索引负责并发兜底，避免业务唯一性被并发击穿。
