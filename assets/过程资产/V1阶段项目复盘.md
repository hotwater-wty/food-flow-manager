# V1 阶段项目复盘

## 1. V1 做成了什么

V1 完成了“膳畅管家”后端项目的最小业务闭环：

```text
用户注册登录 -> 查看桌位 -> 创建预约 -> 到店扫码开台 -> 查看菜品 -> 创建堂食订单 -> 商户处理订单 -> 店员清台释放桌位
```

这个闭环覆盖了用户端、商户端、认证鉴权、桌位状态、预约状态、开台会话、订单状态和员工权限等核心后端能力。

## 2. 关键设计判断

### 2.1 将 V1 范围收敛为最小业务闭环

早期功能设想较大，包含排班、支付、导入导出、通知等能力。最终将 V1 收敛为单门店、无前端、接口测试驱动的主业务闭环，把 Redis、MQ、Docker、复杂权限和排班等放入 V2。

这个判断避免了范围失控，也让每个模块都围绕主链路服务。

### 2.2 引入开台会话承载扫码占座

桌位状态本身无法表达“谁占了桌、是否已下单、是否来源于预约”。因此引入 `dining_session` 作为预约、桌位、订单之间的业务上下文。

开台会话解决了三个问题：

- 用户扫码后尚未下单时，系统仍能记录占座关系。
- 创建订单时能校验订单属于当前用户和当前会话。
- 清台时能围绕一次用餐过程释放桌位并完成订单。

### 2.3 将状态流转放在 Service 层校验

V1 中多个实体都存在状态流转：

- 桌位状态。
- 预约状态。
- 开台状态。
- 订单状态。
- 用户和员工账号状态。

项目没有只依赖 Controller 路径判断，而是在 Service 层重新读取实体并校验业务状态，保证跨模块调用或未来复用时不绕过关键规则。

### 2.4 将认证主体和员工角色拆分

JWT 中使用 `loginType` 区分普通用户和员工，使用 `role` 区分店员和店长。

这样避免把“用户端/商户端身份”和“员工权限等级”混在一个字段里，也方便后续迁移到 Spring Security。

### 2.5 实体内部使用枚举，接口边界使用数字

实体状态字段使用枚举，业务判断更清晰；DTO/VO/API 使用数字 code，便于前后端传输和接口测试。

最终形成约定：

- 实体内部：枚举。
- Service 查询条件：优先传枚举。
- DTO/VO/API：使用 Integer code。

## 3. 暂缓到 V2 的问题

V1 中有些问题已经识别，但为了保证主流程收敛，暂缓到 V2：

- 预约、扫码、下单、清台的并发控制。
- 活跃开台会话的数据库唯一性约束。
- 业务编号生成策略升级。
- 分页查询。
- Redis 缓存账号状态和菜品数据。
- 预约时间窗口和超时自动处理。
- 长时等待提醒。
- 操作日志。
- Docker 化部署。
- Spring Security 权限体系升级。

这些问题不会否定 V1 的业务闭环，但代表项目继续增强的方向。

## 4. 面试表达素材

可以这样概括项目：

> 这是一个餐饮预约与堂食管理后端项目。我在 V1 阶段没有急着堆技术，而是先围绕用户预约、到店开台、堂食点餐、商户处理订单和清台释放桌位打通完整业务闭环。项目中重点处理了桌位、预约、开台会话和订单之间的状态流转，并使用 JWT 区分普通用户、店员和店长权限。

可以重点展开：

- 为什么需要开台会话，而不是只靠桌位状态。
- 为什么状态字段在实体层使用枚举。
- 为什么用户端和商户端路径要分开。
- 为什么店长权限适合先放在拦截器统一处理。
- 为什么 V1 暂不引入 Redis/MQ，而是等主流程稳定后再加。
- V1 中有哪些并发风险，V2 准备如何解决。

## 5. 后端开发流程收获

本阶段完整经历了：

```text
项目定位 -> 范围收敛 -> 业务流程设计 -> 领域模型设计 -> 表设计 -> 接口设计 -> 工程搭建 -> 编码迭代 -> 接口测试 -> 收尾复盘
```

关键收获：

- 需求不是越多越好，必须先形成可交付闭环。
- 状态流转类业务需要先画清实体和状态关系。
- Controller 路径归属不等于 Service 职责归属。
- Service 层应承担业务状态和归属校验。
- 接口测试不是最后一步，而是每个模块完成后的验收手段。
- 过程资产需要分类沉淀，否则对话和排错记录会迅速失控。

## 6. 下一阶段建议

V2 不建议一次性铺开所有增强项。优先顺序建议：

1. 并发控制和活跃会话唯一性约束。
2. 分页查询和接口文档完善。
3. Redis 缓存一个高频查询场景。
4. 定时任务处理预约超时或长时等待提醒。
5. Docker 化部署。
