# V1 阶段项目复盘

本文档用于沉淀“膳畅管家”V1 阶段的项目经历。它不是功能清单，也不是代码说明，而是面向未来复盘和面试表达的材料：说明项目如何从想法收敛到主业务闭环，关键设计为什么这样做，开发中出现过哪些问题，以及 V2 应该从哪里继续推进。

## 1. V1 阶段目标与最终结果

V1 的目标是完成一个可运行、可测试、可讲述的最小业务闭环，而不是一次性覆盖完整餐饮系统。

最终完成的主流程：

```text
用户注册登录
  -> 查看空闲桌位
  -> 创建预约 / 直接扫码占座
  -> 到店扫码开台
  -> 查看启售菜品
  -> 创建堂食订单
  -> 商户处理订单
  -> 店员清台释放桌位
```

这个闭环覆盖了后端项目中比较核心的训练点：

- 多角色认证与权限控制。
- 预约、桌位、开台、订单之间的状态流转。
- 跨模块业务协作和事务控制。
- Service 层业务校验。
- 接口文档和接口测试。
- 从设计文档到编码实现再到收尾复盘的完整过程。

V1 的最终状态是：主业务链路已经打通，基础 CRUD 和关键状态流转可运行；并发控制、缓存、分页、编号策略升级等增强项已明确暂缓到 V2。

## 2. 项目从 0 到 V1 的推进过程

本项目不是直接进入编码，而是先经历了一轮较完整的软件开发生命周期。

推进顺序：

```text
项目定位
  -> 范围收敛
  -> 需求分析
  -> 核心业务流程设计
  -> 领域模型设计
  -> 核心表设计
  -> 接口清单设计
  -> 接口规范设计
  -> 技术选型与依赖规划
  -> 项目包结构设计
  -> SQL 初稿
  -> 工程搭建
  -> 编码迭代
  -> 接口测试
  -> V1 收尾
```

这套过程最大的价值是避免“边想边写导致范围失控”。早期功能设想偏大，包含排班、支付、导入导出、通知、缓存、中间件等内容。如果全部放进一期，会导致业务主线无法收束。最终将 V1 聚焦到单门店、无前端、接口测试驱动的主业务闭环。

在编码阶段，每个模块基本遵循以下节奏：

```text
确认接口和业务规则
  -> 编写实体、DTO、VO、Mapper、Service、Controller
  -> 使用 Apifox 测试正向流程
  -> 审查越权、状态错误、空值、重复数据等异常情况
  -> 修正文档和过程资产
```

这个节奏使项目不是只追求“接口能通”，而是持续检查业务边界和工程质量。

## 3. 核心业务闭环复盘

### 3.1 预约与桌位

用户创建预约时，系统需要同时关心预约记录和桌位状态。预约不是孤立数据，它会影响桌位是否还能被其他用户预约或占用。

V1 中的处理方式是：预约成功后桌位进入已预约状态；预约取消、异常取消或预约到店开台后，再根据业务动作改变桌位状态。

这套方案简单直接，适合 V1。但它也带来一个 V2 问题：现实中预约通常只锁定某个时间段，而不是从预约创建开始长期锁定桌位。因此预约时间窗口和超时处理被放入 V2。

### 3.2 扫码占座与开台会话

用户到店扫码后，不能只把桌位改成等待中。因为系统还需要知道：

- 谁占用了桌位。
- 是否来源于预约。
- 是否已经下单。
- 后续订单属于哪一次用餐过程。
- 清台时应该结束哪一次用餐。

因此 V1 引入 `dining_session` 作为开台会话。它是预约、桌位、用户、订单之间的业务上下文。

开台会话的核心作用是把“桌位状态”从一个单纯标记，升级为可追踪的一次用餐过程。

### 3.3 点餐与订单

用户创建订单前，需要满足几个条件：

- 用户已登录。
- 用户存在当前有效开台会话。
- 开台会话属于当前用户。
- 开台状态允许下单。
- 桌位状态与开台状态一致。
- 菜品存在且处于启售状态。

这一步的关键不是插入订单表，而是确认“用户是否真的处在一个合法的堂食场景中”。因此下单逻辑必须围绕开台会话和桌位状态进行校验。

### 3.4 商户处理订单与清台

商户端订单状态按顺序流转，避免跳级更新。例如订单不能从已下单直接变为已完成。

清台则是主流程的结束点。它不仅释放桌位，还要结束开台会话，并确保关联订单已经处于允许清台的状态。

这说明状态流转类业务不能只看单表。一次业务操作通常会同时影响订单、开台会话和桌位。

## 4. 关键设计决策复盘

### 4.1 为什么 V1 要收敛为最小业务闭环

当时的问题：早期设想包含很多功能，如果全部进入一期，会导致项目长期停留在设计和半成品状态。

可选方案：

- 一期覆盖尽可能多的功能。
- 一期只打通主业务闭环，把增强能力放到 V2。

最终选择：V1 只做餐饮预约与堂食管理的主链路。

适合 V1 的原因：后端求职项目首先要证明能完成业务闭环，而不是堆技术名词。主流程稳定后，再加入 Redis、MQ、Docker、Spring Security 才有明确业务落点。

V2 升级方向：围绕 V1 已暴露的问题补强并发、缓存、分页、编号、部署等能力。

### 4.2 为什么引入 `dining_session`

当时的问题：桌位状态只能说明桌位当前是否空闲、预约、等待或用餐中，但无法表达一次完整的用餐过程。

可选方案：

- 只依靠桌位状态和订单记录。
- 引入开台会话表，记录用户、桌位、预约、开台状态和时间。

最终选择：引入 `dining_session`。

适合 V1 的原因：开台会话让预约开台、非预约占座、下单、清台都围绕同一个业务上下文展开，减少状态散落在多个模块中的风险。

V2 升级方向：增加活跃会话唯一性约束，防止同一用户或同一桌位出现多个活跃会话。

### 4.3 为什么状态字段在实体内部使用枚举

当时的问题：状态字段如果长期使用 `Integer`，业务判断会变成数字比较，可读性差，且容易出现无效状态值。

可选方案：

- 实体、DTO、VO 全部使用 `Integer`。
- 实体内部使用枚举，接口边界使用数字 code。

最终选择：实体内部使用枚举，DTO/VO/API 使用 `Integer` code。

适合 V1 的原因：Service 层判断 `TableStatusEnum.WAITING`、`OrderStatusEnum.PLACED` 比判断 `2`、`0` 更清晰，也减少状态含义混乱。

V2 升级方向：继续统一枚举转换方式，并补充更多参数校验和非法 code 的异常处理。

### 4.4 为什么 `loginType` 和 `role` 要拆分

当时的问题：普通用户、员工、店长都涉及权限，但它们不是同一层概念。普通用户和员工是登录主体类型，店员和店长是员工内部角色。

可选方案：

- 只用一个 role 字段表达所有身份。
- 使用 `loginType` 区分登录主体，使用 `role` 区分员工角色。

最终选择：拆分 `loginType` 和 `role`。

适合 V1 的原因：这样可以清楚表达“这是用户端 token 还是管理端 token”，并在管理端内部继续区分店员和店长。

V2 升级方向：迁移到 Spring Security 时，可以将 `loginType` 映射为认证主体类型，将 `role` 映射为权限标识。

### 4.5 为什么权限先用自定义 JWT 拦截器

当时的问题：项目需要实现用户端和商户端隔离、店长权限控制、账号禁用后旧 token 拦截。

可选方案：

- V1 直接引入 Spring Security。
- V1 使用自定义 JWT 拦截器，后续再升级。

最终选择：V1 使用自定义拦截器。

适合 V1 的原因：当前权限规则较少，自定义拦截器更容易理解和调试，也更适合学习 JWT 的完整流程。

V2 升级方向：当权限规则变复杂后，引入 Spring Security，将路径权限、角色权限和认证上下文交给更标准的安全框架管理。

### 4.6 为什么并发控制暂缓到 V2

当时的问题：预约、扫码、下单、清台都存在并发风险，例如重复开台、重复下单、桌位状态覆盖。

可选方案：

- V1 就加入锁、乐观锁、唯一约束和并发测试。
- V1 先通过事务和状态校验保证单请求正确，把并发增强放到 V2。

最终选择：V1 暂缓复杂并发控制。

适合 V1 的原因：V1 的核心目标是打通业务闭环。如果在主流程尚未稳定时过早引入并发控制，会增加实现复杂度并分散注意力。

V2 升级方向：通过条件更新、乐观锁、唯一索引、并发测试，重点补强预约、扫码、下单、清台等关键状态流转。

## 5. 典型问题与解决过程

### 5.1 普通用户 token 可以访问管理端接口

问题表现：早期拦截器只校验 token 是否有效，没有区分用户端和管理端。

根因：JWT 中缺少清晰的登录主体类型，拦截器也没有按路径校验 `loginType`。

解决方式：在 token 中加入 `loginType`，并在拦截器中区分 `/api/user/**` 和 `/api/admin/**`。普通用户 token 不能访问管理端接口，员工 token 也不能访问用户端接口。

复盘价值：鉴权不能只判断“是否登录”，还要判断“以什么身份登录、是否有权限访问当前资源”。

### 5.2 JWT claim 读取时的类型问题

问题表现：从 JWT claim 中读取 `userId`、`employeeId` 等字段时，直接强转存在类型转换风险，使用 `Object` 又容易产生空指针警告。

根因：JWT 解析后的 claim 类型不一定完全等于业务代码期望的 Java 类型。

解决方式：封装统一读取和转换逻辑，将认证信息收敛到 `LoginContext`，避免在业务代码中到处处理 claim 类型。

复盘价值：外部输入进入系统后，应尽早转换为内部稳定模型。JWT claim 本质上也是外部输入。

### 5.3 接口路径归属与 Service 归属混淆导致循环依赖

问题表现：预约开台和非预约开台的 URL 看起来分别属于预约模块和桌位模块，编码时容易把业务逻辑写入对应 Service，最终引发模块互相依赖。

根因：Controller 路径表达的是用户操作入口，Service 应表达业务状态中心，二者不能简单等同。

解决方式：将创建和维护开台会话的核心逻辑统一放入 `DiningSessionService`，其他模块只作为入口或协作方。

复盘价值：接口属于哪个资源入口，不代表核心业务一定归属于哪个 Service。状态中心应该放在最能承载业务一致性的模块。

### 5.4 状态字段被回滚为 `Integer` 导致枚举判断失效

问题表现：实体状态字段被误回滚为 `Integer` 后，Service 中原本使用枚举判断的代码不再可靠，甚至可能出现比较结果永远为 false 的严重问题。

根因：实体内部状态类型和业务判断方式不一致。

解决方式：恢复实体状态字段的枚举类型，保证系统内部用枚举表达业务状态，只在接口输入输出时转换为数字 code。

复盘价值：类型设计不是形式问题。状态字段一旦类型不统一，业务判断会被破坏。

### 5.5 创建订单时重复菜品与列表下标配对问题

问题表现：创建订单时，如果请求中出现重复菜品 ID，简单使用列表长度判断会误判；如果按列表下标配对菜品和请求明细，也可能产生错配。

根因：数据库查询返回顺序不应作为业务配对依据，请求明细也需要明确重复策略。

解决方式：明确重复菜品策略，使用 `Map<dishId, Dish>` 按菜品 ID 进行匹配，而不是依赖列表下标。

复盘价值：只要业务对象存在唯一标识，就应优先按唯一标识关联，而不是依赖集合顺序。

### 5.6 Service 层需要重复校验业务状态

问题表现：有些状态校验如果只放在 Controller 或上游调用方，未来其他 Service 调用该方法时可能绕过校验。

根因：Service 方法是业务能力边界，不能完全信任调用方已经满足前置条件。

解决方式：涉及状态流转的方法在 Service 层重新读取实体，并校验实体归属、当前状态和目标状态是否合法。

复盘价值：Controller 负责入口参数和路径，Service 负责业务规则。状态流转必须落在 Service 层兜底。

## 6. 当前 V1 的边界与不足

V1 已经能支撑主流程演示，但仍有明确边界：

- 并发控制不足：预约、扫码开台、创建订单、清台在并发场景下仍需条件更新、乐观锁或唯一约束增强。
- 活跃会话缺少数据库强约束：目前主要依赖 Service 查询和校验。
- 编号生成策略较简单：预约编号、开台编号、订单编号后续需要统一升级。
- 列表接口缺少分页：管理端数据量增加后需要分页查询。
- 缓存尚未引入：账号状态、菜品列表等高频访问场景仍直接查库。
- 权限体系仍是自定义实现：适合 V1 学习和演示，但复杂权限建议迁移到 Spring Security。
- 部署方式偏本地：后续需要 Docker 化，降低换环境运行成本。

这些不足不影响 V1 作为最小闭环交付，但它们是 V2 的主要任务来源。

## 7. 面试表达整理

### 7.1 30 秒项目介绍

这是一个基于 Spring Boot 3 和 MyBatis-Plus 的餐饮预约与堂食管理后端项目。V1 阶段我重点完成了从用户预约、到店扫码开台、堂食点餐、商户处理订单到清台释放桌位的完整业务闭环。项目中实现了 JWT 鉴权、多角色权限控制、桌位/预约/开台/订单的状态流转，以及 Service 层业务校验，并使用 Knife4j 和 Apifox 做接口文档与接口测试。

### 7.2 2 分钟业务流程介绍

项目面向单门店餐饮场景。用户注册登录后，可以查看空闲桌位并创建预约；到店后，预约用户可以扫码核验开台，未预约用户也可以直接扫码占座。扫码后系统会创建开台会话，用它记录本次用餐过程中的用户、桌位、预约来源和状态。用户只有在有效开台会话下才能查看菜品并创建堂食订单。订单创建后，商户端员工可以查看订单并按顺序更新订单状态。用餐结束后，店员执行清台操作，系统完成开台会话并释放桌位。

这个项目里我重点处理了几个问题：第一，桌位状态本身不足以承载一次用餐过程，所以引入了开台会话；第二，普通用户、员工、店长的权限需要区分，所以 JWT 中拆分了 `loginType` 和 `role`；第三，状态流转不能只依赖接口路径，必须在 Service 层重新校验实体状态和归属；第四，V1 先保证主业务闭环，像并发控制、缓存、分页、Spring Security 这些增强项放到 V2。

### 7.3 可展开追问的问题清单

面试时可以主动准备这些问题：

- 为什么需要开台会话，而不是只使用桌位状态？
- 预约开台和非预约开台在业务上有什么差异？
- 用户创建订单前需要校验哪些条件？
- JWT 中为什么要拆分 `loginType` 和 `role`？
- 禁用账号后，旧 token 为什么还能访问？你是怎么处理的？
- 自定义 JWT 拦截器和 Spring Security 分别适合什么阶段？
- 实体状态字段为什么使用枚举，而接口传输使用数字？
- 为什么 Service 层还要重复校验状态？
- 创建订单时如何避免菜品列表和请求明细错配？
- V1 中有哪些并发风险？V2 准备如何解决？

## 8. 对后续 V2 的启发

V2 不应该重新设计一套系统，而应该沿着 V1 暴露出的真实问题继续增强。

建议优先级：

1. 补强关键状态流转的并发控制，包括预约、扫码开台、创建订单、清台。
2. 增加活跃开台会话唯一性约束，避免同一用户或桌位出现多个活跃会话。
3. 引入分页查询，优先改造订单、预约、菜品、员工等管理端列表。
4. 升级编号生成策略，形成统一的业务编号组件。
5. 引入 Redis 缓存账号状态或菜品列表，选择一个明确场景做完整缓存一致性练习。
6. 评估 Spring Security 迁移，将当前自定义拦截器中的权限逻辑标准化。
7. 使用 Docker Compose 编排 MySQL、Redis 和后端服务，提升项目展示和部署能力。

V1 的核心收获是：后端项目不是把 Controller、Service、Mapper 写出来就结束。真正需要训练的是业务边界、状态一致性、权限模型、异常路径和阶段性收束能力。
