# 技术实现问题记录

本文档用于记录开发过程中的具体实现问题、错误根因、修复方式和验证方法。定位为技术问题卡片，区别于 `设计与开发问题记录.md` 的设计取舍和面试素材沉淀，也区别于 `开发排错笔记.md` 的旧式错题本。

记录边界：本文档只记录能按“现象、根因、修复、验证”说明的具体问题。工程职责划分、模块边界、状态流转原则等内容优先写入 `工程设计决策记录.md`；工具或框架的系统用法整理到 `专题文档/`。

## 1. Apifox / Knife4j 能访问但接口信息不完整

### 问题类型

接口文档生成与接口测试协作。

### 现象

项目已经能访问 `doc.html` 和 `/v3/api-docs`，Apifox 也可以导入 OpenAPI 数据，但文档中只有路径和基础方法信息，缺少清晰的接口说明、参数说明和响应结构说明。

### 根因

项目只接入了 Knife4j/OpenAPI 依赖，但 Controller、DTO、VO 上缺少 OpenAPI 注解。生成器只能根据 Spring MVC 映射推断基础接口，无法自动补全业务语义。

### 修复方式

按模块补充注解：

- Controller 类使用 `@Tag`
- Controller 方法使用 `@Operation`
- 路径参数使用 `@Parameter`
- DTO 和 VO 类及字段使用 `@Schema`

不要求一次性给全项目补完，建议每完成一个模块同步补该模块注解。

### 验证方式

重启项目后访问 `http://localhost:8080/doc.html` 检查接口说明是否完整；访问 `http://localhost:8080/v3/api-docs` 后在 Apifox 中重新导入，检查接口参数和响应模型是否可识别。

### 经验沉淀

Knife4j/OpenAPI 不是“自动写文档”，而是“基于代码和注解生成文档”。路径能显示只代表集成成功，不代表接口文档已经可维护。

## 2. 员工登录 token 中混用 loginType 和员工角色

### 问题类型

JWT 载荷设计与权限模型。

### 现象

员工登录时曾把员工角色值写入 `loginType`，例如将 `STAFF` 或角色 code 当作登录类型。这样会让后续判断用户端、管理端、店员、店长权限时概念混乱。

### 根因

把“登录主体类型”和“员工权限角色”混为同一个概念。普通用户的 `USER` 是主体类型，员工的 `STAFF`、`MANAGER` 是管理端角色，二者不是同一维度。

### 修复方式

JWT 载荷拆成两个字段：

- `loginType`：`USER` 或 `EMPLOYEE`
- `role`：员工登录时才写入，例如 `STAFF` 或 `MANAGER`

同时增加 `JwtClaimConstants` 管理 claim key，避免字符串散落。

### 验证方式

用户登录后解析 token，应包含 `loginType=USER` 且不包含员工角色。员工登录后解析 token，应包含 `loginType=EMPLOYEE` 和 `role=STAFF/MANAGER`。

### 经验沉淀

权限设计中要区分认证主体和授权角色。主体回答“是谁在登录”，角色回答“这个主体能做什么”。

## 3. JWT claim 读取时直接强转导致潜在类型转换问题

### 问题类型

JWT 解析与 Java 类型转换。

### 现象

拦截器中曾直接使用 `(Long) claims.get(...)` 和 `(LoginTypeEnum) claims.get(...)` 读取 token 数据。IDE 对空值和类型转换给出警告，运行时也可能出现类型转换异常。

### 根因

JWT 解析后的 claim 是通用对象，数字字段不应假设一定是 `Long`。同时 `loginType` 写入 token 时是字符串，例如 `USER`，解析后不会自动变成 `LoginTypeEnum`。

### 修复方式

数字 ID 先按 `Object` 读取，再通过 `toString()` 转为 `Long`。枚举字段先按字符串读取，再用 `valueOf` 转枚举：

```java
Object userIdClaim = claims.get(JwtClaimConstants.USER_ID);
Long userId = userIdClaim == null ? null : Long.valueOf(userIdClaim.toString());

String loginTypeValue = claims.get(JwtClaimConstants.LOGIN_TYPE, String.class);
LoginTypeEnum loginType = LoginTypeEnum.valueOf(loginTypeValue);
```

### 验证方式

分别使用用户 token 和员工 token 请求受保护接口，确认不会因为 claim 类型转换返回 401 或抛出 `ClassCastException`。

### 经验沉淀

JWT claim 是跨系统边界的数据，不应按 Java 内部强类型对象直接强转。进入系统时需要做显式解析、校验和转换。

## 4. LoginContext 中允许部分字段为空的原因和使用边界

### 问题类型

登录上下文设计。

### 现象

`LoginInfo` 中同时包含 `userId`、`employeeId`、`loginType`、`employeeRole`。构造对象时，普通用户请求没有 `employeeId` 和 `employeeRole`，员工请求没有 `userId`，因此担心 builder 填入 null 是否会导致空指针。

### 根因

`LoginInfo` 是“当前登录主体”的统一上下文，不是所有字段在所有登录类型下都有值。不同主体只填充自己相关的字段是正常设计。

### 修复方式

保留一个 ThreadLocal，保存 `LoginInfo` 对象。拦截器根据 token 填入：

- 用户请求：`userId`、`loginType=USER`
- 员工请求：`employeeId`、`loginType=EMPLOYEE`、`employeeRole`

业务层按接口类型读取对应字段，不在用户端读取员工字段。

### 验证方式

用户端接口使用 `LoginContext.getUserId()` 能正常拿到用户 ID。管理端接口使用 `LoginContext.getEmployeeId()` 和 `getEmployeeRole()` 能正常拿到员工信息。请求完成后 `afterCompletion` 清理上下文。

### 经验沉淀

上下文字段允许为空不是问题，关键是要明确字段适用场景，并在拦截器里提前保证请求类型和登录主体匹配。

## 5. 管理端接口只校验 token 有效性导致普通用户可访问管理端

### 问题类型

接口权限边界。

### 现象

早期拦截器只校验 JWT 是否有效并解析 `userId`。这意味着普通用户 token 也可能访问 `/api/admin/tables` 这类管理端接口。

### 根因

认证和授权混在一起，只完成了“token 是否有效”的认证，没有继续判断“当前登录主体是否允许访问该路径”。

### 修复方式

在 token 中加入 `loginType`。拦截器基于路径前缀做端侧隔离：

- `/api/user/**` 只允许 `USER`
- `/api/admin/**` 只允许 `EMPLOYEE`

店长专属路径再检查 `employeeRole == MANAGER`。

### 验证方式

使用普通用户 token 请求管理端接口应返回 403。使用员工 token 请求用户端接口应返回 403。使用员工 token 请求普通管理接口应通过。

### 经验沉淀

JWT 有效不等于有权限。认证解决“你是谁”，授权解决“你能访问什么”。

## 6. 员工注册接口纳入店长权限后需要默认店长初始化数据

### 问题类型

权限闭环与初始化数据。

### 现象

员工注册接口被纳入店长权限后，匿名请求不能再创建员工。如果数据库中没有初始店长账号，就无法登录获取店长 token，也无法继续创建员工。

### 根因

系统权限闭环需要一个初始管理入口。把员工注册做成店长权限是正确的，但必须配套初始化一个可登录的店长账号。

### 修复方式

在初始化 SQL 中预置默认店长账号，并使用真实 BCrypt 密文保存默认密码。当前默认账号用于本地开发初始化，后续可在部署文档中要求首次登录后修改密码。

### 验证方式

初始化数据库后，使用默认店长手机号和密码登录 `/api/admin/auth/login`。登录成功后，使用返回的 token 调用员工注册接口。

### 经验沉淀

权限收紧后要同步检查系统是否还存在合法入口。鉴权设计不仅要考虑接口安全，也要考虑初始化和运维流程。

## 7. 实体 status 字段被回滚为 Integer 导致枚举判断失效

### 问题类型

实体建模与状态判断。

### 现象

部分实体的 `status` 字段被误回滚为 `Integer`，但业务代码仍使用 `TableStatusEnum.FREE`、`UserStatusEnum.NORMAL` 等枚举进行判断。IDE 标红，且如果强行绕过，判断语义会失效。

### 根因

项目约定是“实体内部使用枚举，数据库和接口边界使用数字”。实体字段回滚为 `Integer` 后，业务层枚举判断和字段类型不再一致。

### 修复方式

将实体状态字段恢复为对应枚举类型：

- `User.status` 使用 `UserStatusEnum`
- `DiningTable.status` 使用 `TableStatusEnum`
- `Reservation.status` 使用 `ReservationStatusEnum`
- `Employee.status` 使用 `EmployeeStatusEnum`

VO 层继续使用 `Integer status`，由 `entity.getStatus().getCode()` 转出。

### 验证方式

全局搜索实体 `private Integer status`，确认只存在 VO 中。执行 `mvnw clean test` 强制重新编译。检查业务判断均为枚举与枚举比较。

### 经验沉淀

状态字段最容易在“数据库数字”和“业务枚举”之间混淆。实体层类型必须和业务判断保持一致，否则编译通过也可能存在严重语义错误。

## 8. 预约详情接口未校验归属导致用户可查看他人预约

### 问题类型

水平越权。

### 现象

用户端预约详情接口曾只根据 `reservationId` 查询预约，没有校验该预约是否属于当前登录用户。用户可以通过传入其他预约 ID 查看他人预约信息。

### 根因

把“已登录”误认为“可以访问该业务数据”。JWT 鉴权只证明请求方身份，不会自动限制数据范围。

### 修复方式

用户端预约详情接口调用用户专属方法，查询后校验：

```java
reservation.getUserId().equals(LoginContext.getUserId())
```

管理端预约详情接口使用独立方法，不复用用户端归属限制。

### 验证方式

用户 A 创建预约后，用户 B 使用自己的 token 请求用户 A 的预约详情，应返回业务错误。管理端员工 token 可以通过管理端接口查看预约详情。

### 经验沉淀

涉及“我的数据”的接口必须做归属校验。登录校验、角色校验和数据归属校验是三层不同的安全要求。

## 9. 预约编号固定值导致唯一索引冲突

### 问题类型

业务编号生成。

### 现象

预约创建时曾固定写入预约编号 `1001`。第一次创建成功后，第二次创建会触发 `reservation_no` 唯一索引冲突。

### 根因

业务编号字段具有唯一性，不能使用固定测试值进入真实创建逻辑。即使 V1 暂不引入雪花算法，也需要保证基本不重复。

### 修复方式

V1 暂用时间戳加随机数生成预约编号，并保留后续升级为雪花算法或统一编号生成工具的 TODO。

### 验证方式

连续创建多条预约，确认 `reservation_no` 不重复，数据库唯一索引不报错。

### 经验沉淀

业务编号只要进入数据库唯一索引，就不能再使用固定假数据。临时方案也必须满足唯一性底线。

## 10. 创建预约时误写 checkInTime、取消预约时漏写 cancelTime

### 问题类型

业务时间字段语义。

### 现象

创建预约时曾把预约时间写入 `checkInTime`，取消预约时曾只修改状态，没有写入 `cancelTime`。

### 根因

混淆了预约时间、实际到店时间和取消时间三个不同业务时间点。`reserveTime` 是用户计划到店时间，`checkInTime` 是实际扫码到店时间，`cancelTime` 是取消发生时间。

### 修复方式

创建预约时只写入 `reserveTime`，`checkInTime` 保持为空。扫码到店成功时再写入 `checkInTime`。取消预约时更新状态为已取消，并写入 `cancelTime=now`。

### 验证方式

创建预约后检查数据库：`reserve_time` 有值，`check_in_time` 为空，`cancel_time` 为空。取消预约后检查：状态为已取消，`cancel_time` 有值。

### 经验沉淀

时间字段要对应真实业务事件，不能因为字段名相近就提前填充。否则后续统计、排查和状态流转都会产生误导。

## 11. 列表 VO 组装时使用 Map 批量关联数据，避免 N+1 查询

### 问题类型

列表查询性能与关联数据组装。

### 现象

管理端查看开台会话列表时，`DiningSessionVO` 需要包含会话信息和桌位信息，例如桌号、桌位状态。实现时需要根据每条开台记录的 `tableId` 查询对应桌位。如果在 `stream().map(...)` 中对每条记录单独调用 `diningTableService.getById(tableId)`，逻辑直观，但会产生多次数据库查询。

### 根因

列表数据组装时存在主表和关联表信息组合需求。主表一次查出多条记录后，如果逐条查询关联表，会形成 N+1 查询问题：

```text
1 次查询开台列表 + N 次查询桌位详情
```

数据量小时不明显，但列表增长后会增加数据库访问次数，影响接口响应稳定性。

### 修复方式

先从开台列表中提取去重后的 `tableId`，再批量查询桌位列表并转换为 `Map<Long, DiningTable>`：

```java
List<Long> tableIds = sessionList.stream()
        .map(DiningSession::getTableId)
        .distinct()
        .collect(Collectors.toList());

Map<Long, DiningTable> tableMap = diningTableService.listByIds(tableIds)
        .stream()
        .collect(Collectors.toMap(DiningTable::getId, Function.identity()));
```

组装 VO 时通过 `tableMap.get(session.getTableId())` 获取关联桌位：

```java
sessionList.stream()
        .map(session -> toDiningSessionVO(session, tableMap))
        .collect(Collectors.toList());
```

当前项目中将这部分封装为辅助方法，避免主查询流程被批量关联逻辑干扰。

### 验证方式

执行 `mvnw test` 确认编译和 Spring 上下文启动正常。接口层可通过 Apifox 请求管理端开台会话列表，确认列表能返回会话信息和桌位信息。

后续如果需要进一步验证 SQL 次数，可以开启 MyBatis SQL 日志，确认桌位查询从逐条查询变成一次 `IN (...)` 批量查询。

### 经验沉淀

列表接口中组装关联数据时，不要在循环或 `map` 中逐条查数据库。常见做法是：

```text
主表列表查询 -> 提取关联 ID -> 批量查询关联表 -> 转 Map -> 组装 VO
```

这种方式不是过度设计，而是 Service 层处理列表 VO 关联数据的常规手法。它适合 V1 中不引入复杂联表查询或专门 Mapper XML 的场景，也能避免 N+1 查询问题。
