# 工程设计决策记录

本文档用于记录项目开发过程中形成的工程设计原则、边界划分和方案取舍。它不同于具体 Bug 问题卡，也不同于工具使用专题文档。

## 1. 接口路径归属不等于 Service 归属

### 背景

开台会话存在两种入口：预约用户扫码到店，以及未预约用户扫码占座。从接口路径看，前者容易设计在预约资源下，后者容易设计在桌位资源下。

例如：

```text
POST /api/user/reservations/{reservationId}/check-in/{tableId}
POST /api/user/tables/{tableId}/sessions
```

如果只按路径中的资源名称组织代码，预约开台可能被放入 `ReservationService`，非预约开台可能被放入 `DiningTableService`。

### 冲突点

URL 表达的是用户操作入口，但 Service 表达的是业务用例和状态边界。预约开台和非预约开台的入口不同，但核心业务都是创建和维护 `dining_session`，并推动桌位进入等待中状态。

如果把开台逻辑拆散到预约模块和桌位模块中，后续容易出现：

- `DiningSessionService`、`ReservationService`、`DiningTableService` 相互调用，形成循环依赖。
- 开台事务边界分散。
- 桌位、预约、会话状态流转规则重复或不一致。
- 后续订单模块难以围绕统一的开台上下文继续推进。

### 可选方案

方案一：按 URL 资源归属放置业务逻辑。

预约开台放入预约模块，非预约开台放入桌位模块。实现直观，但会导致开台状态流转分散。

方案二：按业务状态中心放置业务逻辑。

开台相关用例统一放入 `DiningSessionService`。Controller 可以保留不同 URL 入口，但最终调用统一的开台服务。

### 最终决策

采用方案二。

URL 按用户操作入口设计，Controller 按接口入口组织，Service 按业务状态中心组织。

当前项目中，预约开台、非预约开台、查看当前开台、取消等待中开台统一归入 `DiningSessionService`。`ReservationService` 负责预约创建、取消、查询；`DiningTableService` 负责桌位基础资料、启用禁用、查询。

### 适用原则

判断业务逻辑归属时，不只看 URL 中出现了哪个名词，而要看这次操作主要创建或改变哪个核心业务对象。

可按以下规则判断：

```text
URL 按用户从哪里进入设计。
Controller 按接口入口组织。
Service 按谁主导业务状态变化组织。
```

### 当前项目落点

`DiningSessionService` 可以依赖 `ReservationService` 和 `DiningTableService` 来读取和更新预约、桌位数据，但 `ReservationService` 和 `DiningTableService` 不应再反向依赖 `DiningSessionService` 来完成开台流程。

这样可以避免循环依赖，并让开台事务边界集中在一个业务用例中。

## 2. Service 层需要重复校验实体业务状态以保证健壮性

### 背景

在 Controller 层和接口路径设计正确的情况下，请求通常会进入预期的业务方法。但随着模块增多，同一个 Service 方法可能被多个入口调用，也可能在未来被其他业务复用。

对于状态流转类业务，仅依赖调用方已经做过校验是不够的。

### 冲突点

如果 Service 层信任上游调用方，就会减少重复判断，代码看起来更简洁。但状态流转一旦被错误入口调用，或者实体状态在并发、历史数据、未来代码调整中发生变化，就可能绕过关键业务约束。

如果 Service 层重新校验实体状态，会增加一些重复代码，但能让业务方法自身具备更强的防御性和可复用性。

### 可选方案

方案一：只在入口处校验。

Controller 或上游 Service 保证参数和状态正确，底层业务方法直接执行更新。代码较短，但健壮性弱。

方案二：在状态流转的 Service 方法中重新读取实体并校验状态。

每个关键业务方法自己确认实体存在、状态合法、关联关系正确，再执行更新。代码稍多，但边界清晰。

### 最终决策

采用方案二。

涉及业务状态流转的方法，Service 层必须重新校验实体业务状态。入口层校验可以提升用户反馈质量，但不能替代 Service 层的最终防线。

### 适用原则

只要一个方法会改变核心状态，就应在该方法内部校验：

- 实体是否存在。
- 当前状态是否允许本次操作。
- 关联实体状态是否仍然一致。
- 当前登录主体是否有权操作该实体。

这些校验不应完全依赖 Controller 或调用方。

### 当前项目落点

例如取消等待中开台时，不能只根据 `sessionId` 直接把会话取消并释放桌位，而应在 `DiningSessionService` 中校验：

- 开台会话存在。
- 开台会话状态为 `WAITING`。
- 关联桌位存在。
- 关联桌位状态仍为 `WAITING`。
- 释放桌位时同步清空 `current_session_id`。

这样可以避免跨模块调用、并发变化或未来代码复用时绕过关键状态约束。

## 3. 列表 VO 与详情 VO 应按接口场景分离

### 背景

订单查询存在两类接口：管理端订单列表查询，以及管理端订单详情查询。列表查询只需要订单摘要信息，详情查询才需要订单明细列表。

### 冲突点

如果复用同一个 VO，并让 `items` 在列表接口中为空、详情接口中非空，代码可以运行，但接口语义不清晰。调用方需要理解“空列表”到底表示没有明细，还是当前接口不返回明细。

### 最终决策

列表接口和详情接口使用不同 VO。

列表 VO 只表达订单摘要；详情 VO 表达订单摘要加订单明细。不要在同一个 VO 中用“某些字段按场景为空”的方式承载多个接口语义。

### 适用原则

VO 应面向接口场景建模，而不是面向实体字段全集建模。允许为了接口契约清晰而在列表 VO 和详情 VO 之间保留少量字段重复。

### 当前项目落点

管理端订单列表建议使用 `AdminDiningOrderListVO`，管理端订单详情建议使用 `AdminDiningOrderDetailVO`。`items` 只出现在详情 VO 中。

## 4. Java 数字类型应按业务语义和数据库类型统一

### 背景

订单模块开发时，金额字段在不同对象中出现了 `Integer` 和 `Long` 混用，例如订单总金额、菜品单价、明细金额。数据库脚本中这些字段使用的是 `INT`，而主键和外键使用的是 `BIGINT`。

### 最终决策

按业务语义统一数字类型：

- 主键和外键 ID：数据库 `BIGINT`，Java 使用 `Long`。
- 金额、单价、数量、容量、排序：数据库 `INT`，Java 使用 `Integer`。
- 状态：数据库使用数字，实体内部使用枚举，DTO/VO 可使用 `Integer` code。

### 适用原则

不要因为 Java 的 `Long` 容量更大就默认所有数字都用 `Long`。类型应表达业务语义：`Long` 主要表示标识类大整数，`Integer` 主要表示普通数值、金额分、数量和状态码。

### 当前项目落点

订单金额字段统一为 `Integer`，包括 `totalAmount`、`dishPrice`、`amount`。订单 ID、用户 ID、桌位 ID、会话 ID、菜品 ID 继续使用 `Long`。
