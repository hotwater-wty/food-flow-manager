# 技术选型与依赖规划

## 1. 文档目标

本文档用于明确 `food-flow-manager` V1 阶段的技术选型、依赖范围和暂不引入的技术，避免工程搭建时出现技术堆叠或依赖混乱。

V1 的技术原则：

- 优先保证核心业务闭环可运行。
- 优先选择当前阶段能理解、能调试、能讲清楚的技术。
- 不为了简历堆技术点而提前引入复杂组件。
- Redis、MQ、Docker 等增强能力放到 V2，在业务主流程稳定后再接入。

## 2. 项目基础信息

- 项目英文名：`food-flow-manager`
- 项目中文名：膳畅管家
- 项目类型：Java 后端单体项目
- 架构形态：Spring Boot 单体分层架构
- 接口形态：RESTful 风格 HTTP API
- 开发阶段：V1 最小业务闭环

## 3. 核心技术选型

### JDK

建议版本：

```text
JDK 17
```

选择原因：

- Spring Boot 3.x 要求 Java 17 及以上。
- JDK 17 是长期支持版本，稳定性较好。
- 当前 Java 后端项目中使用较广，适合简历项目。

### Spring Boot

建议版本：

```text
Spring Boot 3.x
```

选择原因：

- 用户当前更倾向于 Spring Boot 3.x。
- 生态成熟，适合构建单体后端项目。
- 方便集成 Web、Validation、MyBatis Plus、Knife4j 等能力。

注意事项：

- Spring Boot 3.x 使用 `jakarta.*` 包，不再使用旧版 `javax.*`。
- 搜索资料或参考代码时，需要注意版本差异。

### 构建工具

建议选择：

```text
Maven
```

选择原因：

- Java 后端项目常用。
- 依赖管理清晰。
- 适合当前学习和求职项目。

### Web 框架

核心依赖：

```text
spring-boot-starter-web
```

用途：

- 提供 Controller、REST API、JSON 序列化、内置 Tomcat 等基础能力。

### ORM 与数据库访问

建议选择：

```text
MyBatis Plus
```

核心依赖：

```text
mybatis-plus-spring-boot3-starter
```

选择原因：

- 用户已有 MyBatis 基础，上手成本低。
- 简单 CRUD 可以使用 MyBatis Plus 快速完成。
- 复杂 SQL 仍然可以写 Mapper XML。
- 适合单体后端项目。

### 数据库

建议选择：

```text
MySQL 8.x
```

选择原因：

- Java 后端常见关系型数据库。
- 适合预约、桌位、订单等结构化业务。
- 后续部署和面试讲解都比较自然。

数据库驱动：

```text
mysql-connector-j
```

### 数据库连接池

默认选择：

```text
HikariCP
```

说明：

- Spring Boot 默认使用 HikariCP。
- V1 不需要额外引入 Druid。

### 接口文档

建议选择：

```text
Knife4j / OpenAPI
```

用途：

- 生成项目内置接口文档。
- 支持接口分组、参数说明和在线调试。

说明：

- 日常接口调试可以继续使用 Apifox。
- Knife4j 用于项目自带文档能力。
- Apifox 用于接口集合、环境变量和测试用例管理。

### 登录鉴权

建议选择：

```text
JWT + 自定义拦截器
```

核心依赖可选：

```text
jwt 工具库
```

选择原因：

- 用户已经有 JWT 案例基础。
- V1 权限模型较简单，只区分普通用户、店员、店长。
- 自定义拦截器足够支撑当前项目。

暂不选择：

```text
Spring Security
```

原因：

- Spring Security 功能强，但学习和配置成本较高。
- 当前 V1 不需要复杂权限体系。
- 后续若做权限增强，再考虑引入。

### 参数校验

建议选择：

```text
spring-boot-starter-validation
```

用途：

- 使用 `@NotNull`、`@NotBlank`、`@Min` 等注解校验 DTO 参数。
- 配合全局异常处理器统一返回参数错误信息。

### JSON 处理

默认选择：

```text
Jackson
```

说明：

- Spring Boot Web 默认集成 Jackson。
- V1 不需要额外引入 Fastjson。

### Lombok

建议引入：

```text
lombok
```

用途：

- 简化 Entity、DTO、VO 中的 getter、setter、构造方法等样板代码。

注意事项：

- 需要 IDE 安装或启用 Lombok 支持。
- 不要因为 Lombok 省略对类结构的理解。

### 测试

建议引入：

```text
spring-boot-starter-test
```

用途：

- 单元测试和集成测试。
- V1 可先覆盖关键 Service 层逻辑。

## 4. V1 建议依赖清单

`pom.xml` 中大致需要以下依赖：

```text
spring-boot-starter-web
spring-boot-starter-validation
mybatis-plus-spring-boot3-starter
mysql-connector-j
knife4j-openapi3-jakarta-spring-boot-starter
jwt 工具库
lombok
spring-boot-starter-test
```

说明：

- 具体版本在创建工程时再结合 Spring Boot 版本确定。
- MyBatis Plus 和 Knife4j 需要选择支持 Spring Boot 3.x / Jakarta 的版本。

## 5. V1 暂不引入的技术

### Redis

暂不引入原因：

- V1 先完成主业务闭环。
- 当前验证码、缓存、防重复提交、催单频率控制等场景都放到 V2。

后续可用于：

- 短信验证码存储。
- 菜品和桌位缓存。
- 防重复提交。
- 催单频率控制。

### 消息队列

暂不引入原因：

- V1 订单创建后直接写库即可。
- 异步通知、催单、日志异步落库等场景放到 V2。

后续可选：

- RabbitMQ
- RocketMQ

V2 建议优先选择 RabbitMQ，学习和本地部署成本相对更低。

### Docker

暂不作为 V1 编码阻塞项。

说明：

- V1 可以先本地启动 MySQL 和后端服务。
- Docker 化部署放到 V2 或 V1 末尾整理阶段。

后续可做：

- Dockerfile
- docker-compose 启动 MySQL、Redis、后端服务

### Spring Security

暂不引入原因：

- V1 权限模型简单。
- 使用 JWT + 拦截器即可满足需求。
- Spring Security 可作为后续安全增强方向。

### 在线支付

暂不引入原因：

- V1 不做真实支付。
- 订单创建成功即视为已下单。

后续可扩展：

- 支付宝沙箱
- 微信支付沙箱
- 支付流水表

### 短信服务

暂不引入原因：

- V1 使用手机号 + 密码登录。
- 验证码登录放到 V2。

后续可扩展：

- 阿里云短信服务
- Redis 验证码存储
- 验证码发送频率限制

## 6. 推荐项目配置文件

建议配置文件结构：

```text
src/main/resources
├── application.yml
└── application-dev.yml
```

### application.yml

存放通用配置：

```yaml
spring:
  profiles:
    active: dev

server:
  port: 8080
```

### application-dev.yml

存放本地开发配置：

```yaml
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/food_flow_manager?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: your_password
```

说明：

- 数据库密码不要提交真实线上密码。
- 后续可以通过环境变量管理敏感配置。

## 7. 编码阶段优先落地的技术能力

工程搭建后，建议按以下顺序落地基础能力：

1. Spring Boot 工程启动成功。
2. MySQL 连接成功。
3. MyBatis Plus 基础 CRUD 跑通。
4. 统一响应 `Result<T>`。
5. 全局异常处理器。
6. 参数校验。
7. JWT 生成与解析。
8. 登录拦截器。
9. Knife4j 接口文档。
10. 业务模块接口开发。

## 8. 技术风险与控制

### Spring Boot 3.x 资料差异

风险：

- 网上很多教程基于 Spring Boot 2.x，包名和依赖可能不兼容。

控制：

- 选择支持 Spring Boot 3.x 的依赖版本。
- 注意 `javax.*` 与 `jakarta.*` 的区别。

### JWT 权限边界混乱

风险：

- 用户 Token 和员工 Token 混用。

控制：

- Token 中明确记录登录主体类型。
- 用户端接口只允许普通用户访问。
- 商户端接口只允许员工访问。
- 店长接口额外校验角色。

### 过早引入 Redis / MQ

风险：

- 主业务未完成时引入中间件，增加调试成本。

控制：

- V1 不引入 Redis 和 MQ。
- 先完成业务闭环，再进入技术增强阶段。

## 9. 当前结论

V1 推荐技术组合：

```text
JDK 17
Spring Boot 3.x
Maven
Spring Web
Spring Validation
MyBatis Plus
MySQL 8
JWT + 自定义拦截器
Knife4j / OpenAPI
Lombok
JUnit 5
```

V1 暂不引入：

```text
Redis
MQ
Docker
Spring Security
短信服务
在线支付
```

该组合可以支撑当前 V1 最小业务闭环，也为后续 V2 引入 Redis、MQ、Docker 等技术增强保留空间。
