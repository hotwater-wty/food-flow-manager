# 接口规范设计

## 1. 文档目标

本文档用于约定 V1 阶段后端接口的通用格式和基础规则，避免每个接口各自返回不同结构。

本规范主要覆盖：

- 统一响应格式
- 分页响应格式
- 错误码规则
- 时间格式
- 字段命名规则
- Token 传递方式
- 常见 HTTP 方法使用约定

## 2. 统一响应格式

所有接口统一返回以下结构：

```json
{
  "code": 1,
  "msg": "success",
  "data": {}
}
```

字段说明：

- `code`：业务状态码。`1` 表示成功，`0` 表示失败。
- `msg`：响应描述信息。
- `data`：响应数据。没有数据时可以返回 `null`。

成功示例：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "userId": 1,
    "nickname": "张三"
  }
}
```

失败示例：

```json
{
  "code": 0,
  "msg": "手机号已注册",
  "data": null
}
```

说明：

- HTTP 状态码主要表示请求是否正常到达服务端并被处理。
- `code` 表示业务处理结果。
- V1 先使用 `1` 和 `0` 区分成功失败，后续如果错误类型变多，再扩展更细的业务错误码。

## 3. 分页响应格式

分页接口的 `data` 统一使用以下结构：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "total": 100,
    "pageNo": 1,
    "pageSize": 10,
    "records": []
  }
}
```

字段说明：

- `total`：总记录数。
- `pageNo`：当前页码，从 1 开始。
- `pageSize`：每页条数。
- `records`：当前页数据列表。

分页请求参数统一使用：

- `pageNo`：页码，默认 1。
- `pageSize`：每页条数，默认 10。

请求示例：

```http
GET /api/admin/orders?pageNo=1&pageSize=10&status=1
```

响应示例：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "total": 2,
    "pageNo": 1,
    "pageSize": 10,
    "records": [
      {
        "orderId": 12,
        "orderNo": "O202605260001",
        "tableNo": "A08",
        "totalAmount": 8600,
        "status": 1
      }
    ]
  }
}
```

## 4. 错误码规则

V1 阶段先采用简单规则：

- `1`：成功
- `0`：失败

常见失败信息建议保持明确、可读：

- `手机号已注册`
- `手机号或密码错误`
- `用户未登录`
- `无权限访问`
- `桌位不存在`
- `桌位当前不可预约`
- `预约不存在`
- `预约状态不允许操作`
- `开台会话不存在`
- `开台会话不属于当前用户`
- `菜品不存在或未启售`
- `订单状态不允许变更`
- `系统异常，请稍后重试`

后续可扩展为更细的错误码，例如：

```text
1001 参数校验失败
2001 用户未登录
2002 无权限访问
3001 桌位状态异常
4001 订单状态异常
5000 系统异常
```

但 V1 不急于引入复杂错误码体系，先保证统一异常处理和错误信息清晰。

## 5. 时间格式

接口中的时间字段统一使用字符串格式：

```text
yyyy-MM-dd HH:mm:ss
```

示例：

```json
{
  "reserveTime": "2026-05-26 18:30:00",
  "createTime": "2026-05-26 09:30:00"
}
```

说明：

- V1 默认按系统部署所在时区处理。
- 当前项目开发环境按 Asia/Shanghai 理解时间。
- 日期字段可以使用 `yyyy-MM-dd`，例如排班日期后续可用 `2026-05-26`。

## 6. 字段命名规则

接口 JSON 字段统一使用小驼峰命名：

```json
{
  "reservationId": 1,
  "reservationNo": "R202605260001",
  "reserveTime": "2026-05-26 18:30:00"
}
```

数据库字段使用下划线命名：

```text
reservation_id
reservation_no
reserve_time
```

Java 代码字段使用小驼峰命名：

```java
private Long reservationId;
private String reservationNo;
private LocalDateTime reserveTime;
```

说明：

- 对外接口使用小驼峰，符合 Java 后端常见 JSON 风格。
- 数据库使用下划线，符合 MySQL 表字段常见风格。
- MyBatis Plus 可以负责下划线和驼峰之间的映射。

## 7. Token 传递方式

登录成功后，服务端返回 Token：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "userId": 1,
    "token": "xxxxx.yyyyy.zzzzz"
  }
}
```

调用受保护接口时，前端在请求头中携带：

```http
Authorization: Bearer xxxxx.yyyyy.zzzzz
```

规则：

- 登录、注册接口不需要 Token。
- 用户端受保护接口需要普通用户 Token。
- 商户端受保护接口需要员工 Token。
- 店长专属接口需要校验员工角色为店长。
- 当前登录用户 ID 或员工 ID 由后端从 Token 解析，前端不需要额外传入。

## 8. HTTP 方法约定

V1 接口方法使用原则：

- `GET`：查询数据。
- `POST`：创建资源，或执行带业务动作的操作。
- `PUT`：整体修改资源。
- `DELETE`：删除资源。

示例：

```http
GET /api/user/reservations
POST /api/user/reservations
POST /api/user/reservations/{reservationId}/cancel
PUT /api/admin/tables/{tableId}
DELETE /api/admin/tables/{tableId}
```

说明：

- 像“取消预约”“更新订单状态”“清台释放桌位”这类动作不是简单字段修改，V1 统一用 `POST` 表达业务动作。
- 删除接口在 V1 中可以先做逻辑删除或状态禁用，具体实现阶段再确定。

## 9. 参数校验规则

常见参数校验建议：

- ID 类型字段不能为空，且必须为正数。
- 手机号不能为空，格式需要合法。
- 密码不能为空，长度需要满足最低要求。
- 人数必须大于 0。
- 金额必须大于等于 0。
- 数量必须大于 0。
- 时间字段不能为空时，需要符合统一时间格式。
- 枚举状态必须在允许范围内。

参数校验失败时，统一返回：

```json
{
  "code": 0,
  "msg": "参数校验失败：预约人数必须大于0",
  "data": null
}
```

## 10. 金额格式

接口中的金额统一使用整数，单位为分。

示例：

```json
{
  "dishPrice": 2800,
  "quantity": 2,
  "amount": 5600,
  "totalAmount": 8600
}
```

说明：

- 后端和数据库都使用整数分存储金额。
- 前端展示时再转换为元，例如 `8600` 展示为 `86.00`。
- 这样可以避免浮点数精度问题。

## 11. 空值与空列表

接口返回时遵循以下规则：

- 对象不存在或无数据时，`data` 可以为 `null`。
- 列表查询无数据时，`records` 或列表字段返回空数组 `[]`。
- 可选字段没有值时可以返回 `null`。

示例：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "records": []
  }
}
```

## 12. 后续落地方式

编码阶段建议对应实现以下基础类：

- `Result<T>`：统一响应结果。
- `PageResult<T>`：分页响应结果。
- `BaseException` 或 `BusinessException`：业务异常。
- `ErrorCode` 或 `ResultCode`：错误码枚举，V1 可简单实现。
- `JwtTokenInterceptor`：Token 校验拦截器。
- `GlobalExceptionHandler`：全局异常处理器。

这些基础能力完成后，具体业务接口就可以统一复用，避免每个 Controller 手写不同返回格式。
