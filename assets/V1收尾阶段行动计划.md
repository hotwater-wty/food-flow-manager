# V1 收尾阶段行动计划

## 1. 收尾目标

V1 收尾阶段的目标不是继续扩展业务功能，而是把已经完成的主业务闭环整理为可交付、可复盘、可继续迭代的项目成果。

本阶段围绕五条线推进：

1. 工程验收。
2. 代码与接口收束。
3. V2 待办沉淀。
4. 文档与过程资产整理。
5. Git 收尾。

## 2. 工程验收线

### 2.1 正向主流程验收

使用 Apifox 按以下顺序跑通：

- 用户注册。
- 用户登录。
- 查看空闲桌位。
- 创建预约。
- 预约用户扫码开台。
- 未预约用户扫码占座。
- 查看当前开台。
- 查看启售菜品。
- 创建堂食订单。
- 用户查看订单列表和详情。
- 商户查看订单列表和详情。
- 商户更新订单状态：已下单 -> 制作中 -> 已上齐。
- 店员清台释放桌位。
- 店员释放未下单的等待中桌位。

### 2.2 关键异常验收

至少验证以下异常路径：

- 普通用户 token 不能访问 `/api/admin/**`。
- 员工 token 不能访问 `/api/user/**`。
- 普通员工不能访问员工管理接口。
- 普通员工不能删除桌位。
- 禁用用户或员工后，旧 token 继续请求会被拦截。
- 用户不能查看他人的预约或订单。
- 非等待中的开台会话不能创建订单。
- 桌位状态不是等待中时不能下单。
- 订单状态不能跳级更新。
- 已完成订单不能继续更新状态。

### 2.3 构建验收

执行：

```powershell
.\mvnw.cmd clean test
```

通过后才能进入 Git 合并。

## 3. 代码与接口收束线

### 3.1 README

新增 `README.md`，作为项目对外展示入口，需包含：

- 项目简介。
- 技术栈。
- 核心业务流程。
- 模块说明。
- 本地启动方式。
- 数据库初始化方式。
- 接口文档访问方式。
- V1 已完成功能。
- V2 规划。

### 3.2 接口一致性

以当前代码实现为准，确保 `核心接口清单设计.md` 与 Controller 路径、请求方式保持一致。

当前已经明确的接口标准：

- 商户更新订单状态使用 `PUT /api/admin/orders/{orderId}/status`。
- 预约扫码到店使用 `POST /api/user/reservations/{reservationId}/check-in/{tableId}`。
- 用户订单详情使用 `GET /api/user/orders/{orderId}`。
- 管理端预约详情使用 `GET /api/admin/reservations/{reservationId}/detail`。

### 3.3 V1 开发计划状态

更新 `v1阶段开发计划.md`，明确：

- V1 主业务闭环已完成。
- 管理端权限已补齐到自定义 JWT 拦截器方案。
- 并发控制、分页、缓存、编号策略升级等进入 V2。

## 6. Git 收尾线

当前工作分支：

```text
codex/git-practice-checkpoints
```

本轮默认合并目标：

```text
master
```

原因：本地 `master` 承载已有项目开发历史，远程 `main` 更接近初始分支。

建议流程：

1. 确认工作区只包含 V1 收尾相关改动。
2. 提交 V1 收尾代码与文档。
3. 切换到 `master`。
4. 合并 `codex/git-practice-checkpoints`。
5. 执行 `.\mvnw.cmd clean test`。
6. 推送 `master`。
7. 可选：创建 `v1.0` 标签。
