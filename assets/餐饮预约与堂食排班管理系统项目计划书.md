# 餐饮预约与堂食排班管理系统项目计划书

## 1. 项目定位

本项目拟开发一个面向中小型餐饮商户的预约、堂食订单与员工排班管理系统。

项目不追求覆盖完整餐饮 SaaS 或外卖平台的全部能力，而是聚焦餐饮门店日常经营中更适合后端展示的几个核心场景：用户预约桌位、查看可用座位、到店堂食点餐、商户处理预约与订单、管理员工排班。

该项目的主要目标不是做一个商业级完整产品，而是通过一个有真实业务闭环的系统，沉淀 Java 后端开发能力，并形成可写入简历、可用于面试讲解、可持续扩展的项目资产。

---

## 2. 项目背景

餐饮门店在高峰期通常会面临以下问题：

- 用户无法提前了解门店是否有空位。
- 电话预约和人工登记容易出错。
- 堂食点餐、预约管理和桌位状态之间缺少统一管理。
- 商户需要同时处理预约、订单、桌台和员工排班，管理成本较高。

因此，本项目希望构建一个轻量化的餐饮门店管理系统，帮助用户完成预约与堂食点餐，也帮助商户集中处理预约、桌位、订单和员工排班。

从个人成长角度看，该项目适合作为 Java 后端主项目，因为它既有清晰业务主线，又能自然接入 Redis、MQ、Docker、Linux 部署等后端技术点，复杂度适中，适合当前阶段推进。

---

## 3. 项目目标

### 业务目标

- 为用户提供线上查看空座、预约桌位、到店点餐等基础能力。
- 为商户提供预约处理、桌位管理、堂食订单管理、员工排班等管理能力。
- 构建一条完整的餐饮堂食业务链路。

### 技术目标

- 独立完成一个 Spring Boot 单体后端项目。
- 熟悉后端项目从需求规划、工程搭建、业务开发到部署上线的完整流程。
- 在项目中落地 Redis、消息队列、Docker 等技术，而不是只停留在理论学习。
- 为后续学习微服务保留扩展空间。

### 简历目标

- 形成一个能够写入简历的主项目。
- 能够在面试中讲清项目背景、业务流程、技术选型、核心难点和解决思路。
- 通过项目证明自己具备独立开发 Java 后端项目的能力。

---

## 4. 项目范围

### 一期核心范围

一期版本聚焦“预约 + 堂食 + 商户管理”三条主线。

用户端主要包括：

- 用户注册与登录
- 查看门店基础信息
- 查看桌位和可预约时间段
- 预约桌位
- 到店后创建堂食订单
- 查看个人预约和订单记录

商户端主要包括：

- 预约记录管理
- 桌位状态管理
- 菜品管理
- 堂食订单处理
- 员工基础信息管理
- 员工排班管理

系统基础能力包括：

- 登录鉴权
- 权限区分
- 统一异常处理
- 接口文档
- 日志记录
- Docker 化部署

### 暂不纳入一期范围

以下内容暂不作为一期重点，避免项目范围失控：

- 在线支付
- 完整外卖配送流程
- 骑手系统
- 多门店连锁管理
- 复杂库存管理
- 复杂考勤统计
- 推荐系统
- 完整微服务拆分

这些功能可以作为后续扩展方向，在项目稳定后再逐步考虑。

## 5. 项目价值

### 对用户的价值

- 可以提前查看门店是否有空座。
- 可以在线预约，减少到店等待。
- 可以到店后进行堂食点餐，提升就餐体验。

### 对商户的价值

- 集中管理预约记录，减少人工登记错误。
- 统一管理桌位状态和堂食订单。
- 通过员工排班帮助门店安排日常运营。

### 对个人成长的价值

- 项目业务真实，不是单纯 CRUD。
- 能覆盖登录、权限、订单、状态流转、缓存、消息队列、部署等常见后端能力。
- 便于后续扩展为微服务演示项目。
- 适合包装为实习和校招简历主项目。

---

## 6. 技术规划

项目一期采用单体架构，优先保证业务完整度和工程质量。

计划使用的核心技术包括：

- Java
- Spring Boot
- MyBatis Plus
- MySQL
- Redis
- RabbitMQ 或 RocketMQ
- Docker
- Maven
- Swagger 或 Knife4j

技术使用原则：

- 不为了堆技术而引入技术。
- 每个技术点都要对应明确的业务场景。
- 优先保证项目能跑通、能部署、能讲清楚。
- 微服务相关内容作为后续增强，不作为一期阻塞项。

## 7. 阶段规划

### 第一阶段：项目准备

目标是明确项目方向，完成基础规划。

主要任务：

- 明确项目名称和核心业务范围
- 梳理用户端和商户端主要功能
- 初步拆分核心模块
- 确定技术栈
- 搭建项目仓库和基础工程

阶段成果：

- 项目计划书
- 初步功能清单
- 后端工程骨架

### 第二阶段：核心业务开发

目标是打通最小可用业务链路。

主要任务：

- 完成用户登录与权限区分
- 完成桌位管理
- 完成预约流程
- 完成菜品管理
- 完成堂食订单流程
- 完成商户端基础管理功能

阶段成果：

- 可运行的最小版本
- 用户预约与堂食订单主流程
- 基础接口文档

### 第三阶段：技术增强

目标是在真实业务中落地后端技术点。

主要任务：

- 使用 Redis 优化热点数据访问
- 使用 Redis 或其他方式处理重复提交问题
- 使用消息队列处理异步通知或订单相关任务
- 完善日志、异常处理和接口规范
- 完成 Docker 化部署

阶段成果：

- 具备技术亮点的项目版本
- 可部署说明
- 技术难点记录

### 第四阶段：简历化与面试准备

目标是将项目转化为求职资产。

主要任务：

- 整理 README
- 整理项目架构说明
- 整理核心业务流程说明
- 准备简历项目描述
- 准备面试讲解材料

阶段成果：

- 项目 README
- 简历项目描述
- 面试问答素材

### 第五阶段：后续扩展

目标是在不影响主项目完成度的前提下，增加进阶能力。

可选方向：

- 将通知、订单或用户模块作为微服务拆分演示
- 增加 Nginx 部署
- 增加简单的数据统计
- 增加员工考勤功能
- 增加预约超时取消或提醒机制

## 8. 风险与控制

### 风险一：业务范围过大

控制方式：

- 一期只做预约、堂食订单和排班。
- 外卖、支付、复杂考勤暂不进入一期。
- 每个阶段都以“可运行版本”为目标。

### 风险二：技术点堆叠导致项目失控

控制方式：

- 先完成业务主线，再接入 Redis 和 MQ。
- 每个技术点必须服务于具体业务问题。
- 微服务只作为后续扩展，不影响一期交付。

### 风险三：项目完成但无法用于简历

控制方式：

- 开发过程中同步记录难点和解决方案。
- 每个核心模块都保留可讲解的设计思路。
- 项目后期专门安排简历化和面试准备阶段。

---

## 9. 预期成果

项目完成后，预期形成以下成果：

- 一个可运行的 Java 后端项目。
- 一套清晰的餐饮预约与堂食业务流程。
- 一份完整的项目 README。
- 一份 Docker 部署说明。
- 一份简历项目描述。
- 一组可用于面试讲解的问题和答案。

最终目标是让该项目成为个人 Java 后端学习路线中的主项目，用于沉淀技术基础、补充工程经验，并提升实习投递竞争力。
