# 设计与开发问题记录

本文档用于记录项目设计与开发阶段遇到的问题、思考过程、方案取舍和最终结论。前期以追加方式沉淀素材，后期再统一整理为 README、简历描述和面试问答。

## 1. 接口文档与接口测试工具如何选择

### 问题背景

项目前期计划采用纯后端接口测试方式推进，不做前端页面。可选工具包括 Swagger、Knife4j、Postman、Apifox。当前对 Apifox 更熟悉，但不确定它是否能替代 Swagger/Knife4j。

### 方案思考

- Apifox 适合日常接口调试、环境变量管理、接口集合管理和测试用例沉淀。
- Knife4j/OpenAPI 更适合作为后端项目内置接口文档，能和 Spring Boot 工程绑定，从代码注解生成文档。
- Postman 可以被 Apifox 替代，不是当前项目必须项。

### 当前结论

项目内集成 Knife4j 或 OpenAPI，用于生成随项目维护的接口文档；日常接口调试和测试用例管理使用 Apifox。暂不使用 Postman。

### 面试表达素材

在项目中没有单纯依赖外部接口测试工具，而是将接口文档能力集成进后端工程，保证接口定义、调试和文档维护之间的一致性。同时使用 Apifox 管理接口测试流程，提高本地开发和回归测试效率。

## 2. 一期功能范围过大如何收敛

### 问题背景

最初的一期功能清单包含短信验证码、支付、购物车、催单、csv 导入导出、排班、复杂提醒等能力，容易导致项目范围失控。

### 方案思考

- 先保证核心业务闭环，再做技术增强。
- 将一期拆成 V1 最小业务闭环和 V2 业务增强。
- V1 聚焦预约、到店、开台、点餐、订单处理、清台。
- V2 再引入 Redis、定时任务、MQ、Docker、导入导出、排班等增强内容。

### 当前结论

V1 目标是打通“用户预约桌位 -> 到店开台 -> 堂食点餐 -> 商户处理订单 -> 结束用餐并释放桌位”的主流程。V2 在 V1 稳定后再补充技术亮点和真实业务增强。

### 面试表达素材

项目初期主动控制范围，没有为了堆功能直接引入所有设想，而是按 MVP 思路拆分版本。先实现可运行、可测试、可讲解的主业务闭环，再逐步引入缓存、异步、定时任务和部署能力，降低项目烂尾风险。

## 3. 扫码占座后等待中桌位是否应该自动释放

### 问题背景

用户到店扫码后，系统将桌位从空闲或已预约更新为等待中。这样可以避免线下用户已经坐下，而线上用户又预约到同一桌位的冲突。但如果用户扫码后不下单并自行离开，桌位会长期处于等待中，影响后续使用。

### 方案思考

- 如果设置等待中自动释放，系统可能误释放仍有用户实际占座的桌位，引发线下纠纷。
- 如果频繁弹窗让用户确认是否继续等待，用户体验较差，也无法防止恶意或无响应情况。
- 如果将长时等待用户归入预约状态，会混淆“未来预约占用”和“当前现实占用”两个概念。
- 新增特殊桌位状态会增加状态机复杂度，不利于 V1 实现和讲解。

### 当前结论

保留四种桌位状态：空闲、已预约、等待中、用餐中。

等待中表示用户已经到店并实际占座，但尚未下单。等待中桌位不自动释放。系统可以根据等待时长向商户端提示“长时间未下单”，但是否释放桌位必须由店员线下确认后手动处理。

### 面试表达素材

桌位状态设计时区分了线上资源占用和线下现实占用。已预约表示未来占用，等待中表示用户已经到店并实际占座。对于等待中长时间未下单的问题，系统不直接自动释放，因为系统无法准确判断用户是否仍在现场，而是采用“系统提醒 + 人工确认”的方式，兼顾业务严谨性和现实可操作性。

## 4. 为什么需要引入开台 DiningSession 模型

### 问题背景

在初版领域模型中，已经有用户、桌位、预约、订单、订单明细等对象。但业务流程里存在一个中间状态：用户扫码成功后已经占座，但还没有下单。此时如果只依赖订单表，就无法记录“是谁正在等待中占用这张桌位”。

### 方案思考

- 只在桌位表记录状态，可以知道桌位是等待中，但不知道当前占用人、扫码时间、是否来自预约。
- 只依赖预约表，无法表达未预约用户扫码占座的情况。
- 只依赖订单表，无法表达尚未下单的等待中状态。
- 引入开台会话可以统一承接预约到店和未预约扫码占座两种场景。

### 当前结论

新增 `dining_session` 开台表，用于记录一次真实到店占座过程。预约用户扫码和未预约用户扫码都会创建开台记录。订单关联开台记录，桌位通过 `current_session_id` 指向当前占用会话。

### 面试表达素材

项目中没有把桌位状态简单做成一个枚举字段就结束，而是进一步分析了状态背后的业务归属问题。由于等待中状态下可能还没有订单，因此引入开台会话模型记录实际占座过程，解决了“桌位被谁占用、何时开始等待、是否来源于预约、后续订单归属于哪次用餐”的问题。

## 5. 主键 ID 与业务编号如何取舍

### 问题背景

核心表设计中存在大量“主键 ID + 唯一业务编号”的组合，例如 `dining_session.id` 和 `dining_session.session_no`。需要明确接口返回、前端展示和后端查询时分别应该使用哪个标识。

### 方案思考

- 主键 ID 更适合数据库关联、后端查询、更新和表之间建立关系。
- 业务编号更适合前端展示、用户沟通、店员识别、日志排查和后续可能出现的票据场景。
- 创建成功类接口可以同时返回 ID 和业务编号：ID 供后续接口调用，业务编号供页面展示。
- 当前项目是单体应用，不需要一开始就引入雪花 ID 替代数据库自增主键。

### 当前结论

V1 采用数据库自增 Long 作为主键 ID，另行为预约、开台、订单等核心业务对象生成唯一业务编号。

内部关联和接口操作优先使用 ID；对外展示和人工识别使用业务编号。

### 面试表达素材

项目中区分了数据库主键和业务编号的职责。主键 ID 用于系统内部关联和高效查询，业务编号用于前端展示、用户沟通和日志排查。考虑到项目当前是单体架构，没有过早引入分布式 ID，而是选择自增主键加业务编号的方式，保证实现简单、调试方便，同时保留后续演进空间。

## 6. 用户下单接口应该围绕用户、桌位还是开台会话设计

### 问题背景

用户下单前需要满足多个条件：用户已登录、已经线下扫码占座、桌位状态为等待中、当前用户确实是该桌位本次开台的占用者。最初容易从“用户提交订单”的角度出发设计接口，但这样无法充分表达堂食业务里的桌位和开台上下文。

### 方案思考

- 登录状态由 JWT 鉴权先完成，业务层可以直接拿到当前登录用户。
- 下单动作表面上由用户发起，但业务锚点是“某张桌位当前这次开台会话正在点单”。
- 只围绕用户设计，无法有效防止未占座用户直接下单。
- 只围绕桌位设计，能表达“几号桌点单”，但桌位会被反复使用，无法精确区分是哪一次用餐。
- 围绕开台会话设计最准确，因为 `dining_session` 表示一次具体的到店占座和用餐过程。

### 当前结论

创建堂食订单接口优先围绕开台会话设计，例如：

```http
POST /api/user/sessions/{sessionId}/orders
```

核心校验规则：

- 用户必须已登录。
- 开台会话必须存在。
- 开台会话 `user_id` 必须等于当前登录用户 ID。
- 开台会话状态必须为等待中。
- 桌位状态必须为等待中。
- 菜品必须处于启售状态。
- 校验通过后创建订单和订单明细。
- 更新开台状态为用餐中。
- 更新桌位状态为用餐中。

### 面试表达素材

下单接口没有简单设计成“用户创建订单”，而是围绕开台会话建模。因为堂食订单依赖具体桌位和本次到店占座过程，开台会话能够同时关联用户、桌位、预约和订单，避免未占座下单、替他人桌位下单、重复下单等问题。这体现了接口设计从“操作人”转向“业务上下文”的思考。

## 7. 接口路径如何区分用户端与商户端

### 问题背景

核心接口设计中同时存在用户端接口和商户端接口。如果路径命名混乱，例如所有接口都直接放在 `/api` 下，后续鉴权、权限控制、接口文档分组和前端调用都会变得不清晰。

### 方案思考

- 用户端接口面向普通顾客，主要处理预约、扫码开台、点餐、查看个人记录。
- 商户端接口面向店员和店长，主要处理桌位、预约、菜品、订单和员工管理。
- 登录后的用户 ID 或员工 ID 应由 Token 解析得到，不应作为普通业务参数由前端传入。
- 接口路径需要体现端侧和资源归属，避免用户端和商户端职责混在一起。

### 当前结论

V1 接口统一约定：

- 用户端接口使用 `/api/user` 前缀。
- 商户端接口使用 `/api/admin` 前缀。
- 用户认证接口使用 `/api/user/auth`。
- 员工认证接口使用 `/api/admin/auth`。
- 当前登录主体从 Token 中解析，接口参数只传业务所需字段。

### 面试表达素材

项目接口设计时按端侧划分路径前缀，用户端和商户端分别使用 `/api/user` 和 `/api/admin`。这样做能让接口权限边界更清晰，也方便后续在拦截器、接口文档和前端调用层面做分组管理。同时避免让前端直接传当前用户 ID，降低越权风险。

## 8. 为什么需要先设计接口规范

### 问题背景

核心接口清单只说明了接口路径、权限、参数和业务规则，但还没有统一响应结构、分页格式、错误码、时间格式、Token 传递方式等通用规则。如果没有这些约定，后续编码时每个接口可能返回不同结构，接口文档和前端调试都会变混乱。

### 方案思考

- 接口清单解决“有哪些接口、接口做什么”的问题。
- 接口规范解决“所有接口应该遵守什么统一格式”的问题。
- 详细接口文档后续可以通过 DTO 和 Knife4j/OpenAPI 注解生成，不需要前期纯手工维护所有 JSON 示例。

### 当前结论

V1 先制定轻量接口规范：统一返回 `code/msg/data`，分页返回 `total/pageNo/pageSize/records`，时间使用 `yyyy-MM-dd HH:mm:ss`，JSON 字段使用小驼峰，Token 通过 `Authorization: Bearer xxx` 请求头传递。

### 面试表达素材

项目在编码前先约定接口规范，统一响应结构、分页格式、时间格式和鉴权方式。这样可以降低 Controller 层重复代码，也方便后续通过全局异常处理器、统一返回对象和 Knife4j 文档生成能力保持接口风格一致。

## 9. 数据库是否需要建立物理外键

### 问题背景

核心表之间存在大量关联关系，例如预约关联用户和桌位，开台会话关联用户、桌位和预约，订单关联开台会话和订单明细。设计 SQL 脚本时需要决定是否使用数据库物理外键约束。

### 方案思考

- 物理外键可以在数据库层保证关联完整性，适合规则稳定、删除较少的系统。
- 物理外键会增加表删除、数据初始化、测试数据清理和后续业务调整的复杂度。
- 当前项目是学习型单体项目，业务规则会在编码阶段继续迭代。
- Java 后端项目中也常见“逻辑外键”方案：字段保留关联 ID，数据库建索引，完整性由 Service 层业务规则保证。

### 当前结论

V1 SQL 初稿暂不建立物理外键，只保留关联字段和必要索引。业务层负责校验用户、桌位、预约、开台会话、订单之间的关系。

### 面试表达素材

项目表结构中保留了清晰的关联 ID 和索引，但 V1 没有强制建立数据库物理外键。这样做是为了降低早期开发、测试数据维护和业务迭代成本，同时把状态流转和关联校验放在 Service 层统一控制。后续如果业务模型稳定，也可以再评估是否补充物理外键。
