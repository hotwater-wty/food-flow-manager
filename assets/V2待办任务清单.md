# V2 待办任务清单

本文档用于承接 V1 中已明确暂缓的问题和代码 TODO。它不是完整 V2 需求文档，而是 V2 开发前的候选任务池。

优先级说明：

- P0：影响主链路可靠性，V2 优先处理。
- P1：能明显提升工程质量或项目展示价值。
- P2：增强体验或补充完整性，可排在后面。

## 1. 关键业务流程并发控制

### 问题背景

预约、扫码开台、创建订单、清台都会同时修改多张表和状态字段。如果多个请求并发进入，可能出现重复预约、重复开台、重复下单或桌位状态覆盖。

### 当前 V1 处理方式

V1 使用事务保证单次请求内的数据一致性，并在 Service 层校验实体状态，但没有使用行锁、乐观锁或分布式锁。

### V2 目标

保证关键状态流转在并发场景下仍然可靠。

### 建议技术方案

- 对桌位状态流转使用条件更新，例如 `where id = ? and status = ?`。
- 对关键表增加 `version` 字段实现乐观锁。
- 对创建预约、扫码开台、创建订单等接口补充并发测试。
- 如引入 Redis，可评估基于业务 key 的短时锁。

### 优先级

P0

## 2. 活跃开台会话唯一性约束

### 问题背景

业务上一个用户同一时间只应有一个等待中或用餐中的开台会话。若出现脏数据或并发创建，`getCurrentSession()` 查询活跃会话时可能查出多条。

### 当前 V1 处理方式

Service 层在开台前检查用户是否已有活跃会话，并使用 `.one()` 查询当前会话，但数据库层还没有强约束。

### V2 目标

从业务层和数据库层共同约束“一个用户最多一个活跃开台会话”。

### 建议技术方案

- 增加 `active_flag` 字段，等待中/用餐中为 1，完成/取消为 0。
- 建立唯一索引 `(user_id, active_flag)`，并只让活跃状态占用唯一约束。
- 更新开台完成或取消逻辑时同步维护 `active_flag`。

### 优先级

P0

## 3. 编号生成策略升级

### 问题背景

预约编号、开台编号、订单编号目前使用简单生成策略。V1 可用，但在高并发、分布式或长期运行场景下不够稳健。

### 当前 V1 处理方式

使用工具类生成业务编号，并依赖数据库唯一索引兜底。

### V2 目标

形成统一、可解释、冲突概率低的业务编号生成方案。

### 建议技术方案

- 单体项目可先使用时间戳 + 随机数 + 数据库唯一索引重试。
- 如果需要展示技术亮点，可引入雪花算法。
- 将预约、开台、订单编号统一收敛到编号生成组件。

### 优先级

P1

## 4. 分页查询

### 问题背景

当前多个列表接口直接返回全量数据，例如桌位、预约、开台会话、菜品、订单、员工等。数据量增加后会影响性能和接口体验。

### 当前 V1 处理方式

V1 为降低复杂度，优先返回列表。

### V2 目标

管理端列表接口支持分页查询，必要时支持筛选条件。

### 建议技术方案

- 使用 MyBatis-Plus 分页插件。
- 统一 `PageResult` 返回结构。
- 优先改造订单列表、预约列表、菜品列表、开台会话列表。

### 优先级

P1

## 5. 用户端菜品详情

### 问题背景

V1 主流程已支持用户查看启售菜品列表，但用户端菜品详情接口仍可补齐。

### 当前 V1 处理方式

用户端主要通过菜品列表完成点餐，详情不是主流程必需。

### V2 目标

支持用户查看启售菜品详情，并禁止查看停售或不存在的菜品。

### 建议技术方案

- 新增 `GET /api/user/dishes/{dishId}`。
- Service 层校验菜品存在且 `ON_SALE`。
- 返回字段复用 `DishVO`。

### 优先级

P2

## 6. 账号状态缓存

### 问题背景

当前拦截器每次请求都会查询用户或员工账号状态，保证禁用后旧 token 立即失效，但会增加数据库访问。

### 当前 V1 处理方式

直接查数据库，逻辑简单可靠。

### V2 目标

在不降低权限实时性的前提下减少数据库查询。

### 建议技术方案

- 引入 Redis 缓存账号状态。
- 禁用或启用账号时同步更新缓存。
- 缓存未命中时回源数据库。

### 优先级

P1

## 7. 菜品缓存

### 问题背景

启售菜品列表属于高频查询，适合缓存。

### 当前 V1 处理方式

每次请求直接查询数据库。

### V2 目标

降低菜品列表查询压力，并练习 Redis 缓存一致性。

### 建议技术方案

- 使用 Redis 缓存启售菜品列表。
- 菜品新增、修改、删除、状态变更时删除或刷新缓存。
- 可按分类维度缓存。

### 优先级

P1

## 8. 预约时间策略优化

### 问题背景

V1 中预约成功后桌位会直接进入已预约状态，简单但不够贴近真实业务。现实中桌位可能在预约时间前仍可被其他顾客使用。

### 当前 V1 处理方式

预约创建后桌位进入 `RESERVED`，直到到店、取消或异常处理。

### V2 目标

让预约时间策略更接近真实餐饮业务。

### 建议技术方案

- 引入预约保留时长和可提前预约范围配置。
- 预约只锁定目标时间窗口。
- 通过定时任务处理超时预约。
- 商户端显示即将到店或超时预约。

### 优先级

P1

## 9. 长时等待提醒

### 问题背景

用户扫码占座后可能长时间不下单。V1 不自动释放桌位，符合“人坐在那里就代表占用”的业务判断，但需要提醒店员人工确认。

### 当前 V1 处理方式

店员通过释放等待中桌位接口手动处理。

### V2 目标

系统自动识别长时等待并提醒商户端，但不自动释放现实占用桌位。

### 建议技术方案

- 定时任务扫描等待中开台会话。
- 超过阈值后标记提醒状态或写入操作日志。
- 商户端列表增加等待时长或提醒标识。

### 优先级

P1

## 10. 操作日志

### 问题背景

桌位、预约、订单、员工等模块存在较多状态变化。缺少操作日志会影响排查和面试展示。

### 当前 V1 处理方式

主要依赖应用日志和数据库状态。

### V2 目标

关键业务操作落库，支持按类型或对象查询。

### 建议技术方案

- 设计 `operation_log` 表。
- 记录操作人、操作类型、业务对象、前后状态、时间。
- 可先同步落库，后续再使用 MQ 异步化。

### 优先级

P1

## 11. Spring Security 权限体系升级

### 问题背景

V1 使用自定义 JWT 拦截器完成用户端/商户端隔离和店长权限控制。规则简单时足够，但复杂权限会让拦截器膨胀。

### 当前 V1 处理方式

自定义 `JwtTokenInterceptor` 解析 token、校验账号状态、校验路径权限。

### V2 目标

学习并评估 Spring Security 在项目中的落地方式。

### 建议技术方案

- 使用 Spring Security Filter 替代自定义拦截器。
- 将员工角色映射为 GrantedAuthority。
- 使用注解或配置控制店长权限。
- 保留现有 JWT 签发逻辑或逐步迁移。

### 优先级

P2

## 12. Docker 化部署

### 问题背景

当前项目依赖本地 MySQL 和本地配置。对外展示或换机器运行时成本较高。

### 当前 V1 处理方式

手动安装 MySQL，手动执行 SQL，修改本地配置文件。

### V2 目标

使用 Docker 提供更稳定的本地部署流程。

### 建议技术方案

- 编写 Dockerfile。
- 使用 docker-compose 编排 MySQL、Redis 和后端服务。
- 将数据库账号密码等敏感配置改为环境变量。

### 优先级

P2

## 13. 消息队列与异步通知

### 问题背景

订单创建、催单、预约提醒、操作日志等场景适合异步处理，但 V1 没有引入 MQ。

### 当前 V1 处理方式

请求内同步完成所有必要业务操作。

### V2 目标

选择一个明确业务场景落地 MQ，避免为堆技术而全面改造。

### 建议技术方案

- 优先选择订单创建后通知商户端或操作日志异步落库。
- 可使用 RabbitMQ。
- 保证失败重试和日志可观测。

### 优先级

P2
