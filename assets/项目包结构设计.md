# 项目包结构设计

## 1. 设计目标

本文档用于规划 `food-flow-manager` 后端工程的包结构，目标是让 V1 单体项目在编码阶段保持清晰、可维护、便于扩展。

V1 采用 Spring Boot 单体分层架构，不做微服务拆分。

## 2. 基础包名

建议基础包名：

```text
com.foodflow
```

主启动类：

```text
com.foodflow.FoodFlowManagerApplication
```

说明：

- `foodflow` 对应项目英文名 `food-flow-manager`。
- 包名保持简短，避免后续类路径过长。

## 4. 公共层说明

### common

存放跨模块复用的通用代码。

```text
common
├── constant
├── enums
├── exception
├── result
└── utils
```

#### common.constant

存放通用常量。

示例：

```text
JwtClaimsConstant
MessageConstant
RedisKeyConstant
```

V1 暂不引入 Redis，因此 `RedisKeyConstant` 可以后续再加。

#### common.enums

存放通用枚举。

示例：

```text
UserStatusEnum
EmployeeStatusEnum
EmployeeRoleEnum
TableStatusEnum
ReservationStatusEnum
DiningSessionStatusEnum
DishStatusEnum
OrderStatusEnum
```

说明：

- 状态类字段不要在业务代码里直接写魔法数字。
- 枚举可以帮助统一状态含义。

#### common.exception

存放异常相关类。

示例：

```text
BusinessException
GlobalExceptionHandler
```

说明：

- 业务异常统一抛 `BusinessException`。
- 全局异常处理器统一转换为 `Result` 响应。

#### common.result

存放统一响应结果。

示例：

```text
Result<T>
PageResult<T>
```

对应前面接口规范里的：

```json
{
  "code": 1,
  "msg": "success",
  "data": {}
}
```

#### common.utils

存放工具类。

示例：

```text
JwtUtil
PasswordUtil
NumberGenerator
```

说明：

- `JwtUtil` 用于生成和解析 Token。
- `PasswordUtil` 用于密码加密与校验。
- `NumberGenerator` 用于生成预约编号、开台编号、订单编号。

## 5. 配置与拦截层说明

### config

存放 Spring Boot 配置类。

示例：

```text
WebMvcConfig
MybatisPlusConfig
Knife4jConfig
```

说明：

- `WebMvcConfig` 用于注册拦截器、跨域配置等。
- `MybatisPlusConfig` 用于分页插件等配置。
- `Knife4jConfig` 用于接口文档配置。

### interceptor

存放请求拦截器。

示例：

```text
JwtTokenInterceptor
```

说明：

- 校验请求头中的 Token。
- 解析当前登录用户或员工信息。
- 将当前登录主体放入上下文，供业务层获取。

### security

存放和登录鉴权上下文相关的代码。

示例：

```text
LoginUser
LoginEmployee
BaseContext
```

说明：

- V1 不一定引入 Spring Security。
- 可以先使用自定义 JWT 拦截器加 ThreadLocal 保存当前登录主体。
- 后续如果引入 Spring Security，再调整该包职责。

## 6. 业务模块说明

V1 采用按业务模块划分包的方式：

```text
module.user
module.employee
module.table
module.reservation
module.session
module.dish
module.order
```

这种方式比按技术层统一放置更适合中等规模项目，因为每个业务模块的 Controller、Service、Mapper、DTO、VO 都集中在一起。

## 7. 模块内部结构说明

每个业务模块内部建议保持一致结构：

```text
controller
service
service.impl
mapper
entity
dto
vo
```

### controller

负责接收请求、参数校验、调用 Service、返回结果。

原则：

- 不写复杂业务逻辑。
- 不直接操作 Mapper。
- 不直接拼接复杂返回数据。

示例：

```text
UserAuthController
UserReservationController
AdminOrderController
```

说明：

- 用户端和商户端接口可以通过 Controller 命名区分。
- 例如用户端订单接口叫 `UserOrderController`，商户端订单接口叫 `AdminOrderController`。

### service

存放 Service 接口。

示例：

```text
ReservationService
DiningSessionService
DiningOrderService
```

### service.impl

存放 Service 实现类。

示例：

```text
ReservationServiceImpl
DiningSessionServiceImpl
DiningOrderServiceImpl
```

原则：

- 核心业务规则写在 Service 层。
- 涉及多个表的状态变更，应该放在 Service 层并使用事务。
- 例如创建预约、扫码开台、创建订单、清台释放桌位。

### mapper

存放 MyBatis Plus Mapper。

示例：

```text
UserMapper
ReservationMapper
DiningOrderMapper
```

原则：

- Mapper 只负责数据库访问。
- 简单 CRUD 优先使用 MyBatis Plus 提供的方法。
- 复杂 SQL 再写 XML 或自定义方法。

### entity

存放数据库实体类。

示例：

```text
User
Employee
DiningTable
Reservation
DiningSession
Dish
DiningOrder
OrderItem
```

原则：

- Entity 对应数据库表。
- 不直接作为接口请求参数。
- 不建议直接返回给前端。

### dto

存放请求参数对象。

示例：

```text
UserRegisterDTO
UserLoginDTO
CreateReservationDTO
CreateOrderDTO
UpdateOrderStatusDTO
```

说明：

- DTO 表示前端传给后端的数据。
- 参数校验注解通常写在 DTO 上。

### vo

存放响应视图对象。

示例：

```text
UserLoginVO
ReservationVO
DiningSessionVO
OrderDetailVO
TableVO
```

说明：

- VO 表示后端返回给前端的数据。
- VO 可以组合多个表的数据。
- 避免直接把 Entity 暴露给前端。

## 8. 各模块职责建议

### module.user

负责普通用户相关能力：

- 用户注册
- 用户登录
- 用户基础信息

### module.employee

负责员工相关能力：

- 员工登录
- 新增员工
- 启用员工
- 禁用员工
- 查看员工列表

### module.table

负责桌位相关能力：

- 桌位新增、修改、删除、查询
- 桌位状态查询
- 桌位释放操作可以由 `session` 模块承接，`table` 模块只提供基础桌位能力。

### module.reservation

负责预约相关能力：

- 创建预约
- 查看我的预约
- 用户取消预约
- 商户查看预约
- 商户取消异常预约
- 预约用户扫码到店可以调用 `session` 模块创建开台会话。

### module.session

负责开台会话相关能力：

- 预约用户扫码到店创建开台
- 未预约用户扫码占座创建开台
- 查询当前开台
- 清台释放桌位
- 释放未下单的等待中桌位

说明：

- `session` 是本项目比较关键的业务模块。
- 它承接“线下现实占座”这一概念。

### module.dish

负责菜品相关能力：

- 菜品分类查询
- 菜品新增、修改、删除
- 菜品状态变更
- 用户查看启售菜品

### module.order

负责订单相关能力：

- 创建堂食订单
- 查看用户订单
- 商户查看订单
- 商户更新订单状态

说明：

- 创建订单时需要依赖 `session` 模块校验开台状态。
- 创建订单成功后需要更新开台状态和桌位状态。

## 9. 命名建议

### 类命名

```text
XxxController
XxxService
XxxServiceImpl
XxxMapper
XxxDTO
XxxVO
```

示例：

```text
UserAuthController
ReservationService
ReservationServiceImpl
ReservationMapper
CreateReservationDTO
ReservationVO
```

### 接口端侧命名

用户端 Controller 建议加 `User` 前缀：

```text
UserReservationController
UserOrderController
```

商户端 Controller 建议加 `Admin` 前缀：

```text
AdminReservationController
AdminOrderController
```

说明：

- 这样能直观看出接口归属。
- 也方便 Knife4j 接口文档分组。

## 10. 资源目录建议

```text
src/main/resources
├── application.yml
├── application-dev.yml
├── mapper
│   ├── user
│   ├── employee
│   ├── table
│   ├── reservation
│   ├── session
│   ├── dish
│   └── order
└── db
    └── schema.sql
```

说明：

- `application.yml` 放公共配置。
- `application-dev.yml` 放本地开发配置。
- `mapper` 存放 MyBatis XML 文件。
- `db/schema.sql` 存放建表脚本。

## 11. V1 建议优先实现的基础类

工程搭建后，可以优先创建这些基础类：

```text
common.result.Result
common.result.PageResult
common.exception.BusinessException
common.exception.GlobalExceptionHandler
common.enums.TableStatusEnum
common.enums.ReservationStatusEnum
common.enums.DiningSessionStatusEnum
common.enums.OrderStatusEnum
common.utils.JwtUtil
interceptor.JwtTokenInterceptor
security.BaseContext
```

这些类完成后，再进入具体业务模块开发会更顺。

## 12. 注意事项

- 不要把所有 Controller 都放在一个包里，否则用户端和商户端会混乱。
- 不要直接把 Entity 作为接口返回对象。
- 不要让 Controller 直接调用 Mapper。
- 涉及多个表状态变化的操作需要加事务。
- 业务状态不要在代码里直接写数字，优先使用枚举或常量。
- V1 不做微服务拆分，先保证单体项目结构清晰。
