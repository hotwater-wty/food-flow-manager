# V2 业务增强功能清单

## 1. 版本目标

V2 的目标是在 V1 最小业务闭环稳定的基础上，补充更接近真实业务的增强能力，并逐步引入 Redis、定时任务、消息队列、Docker 等后端技术点。

V2 不应推翻 V1 的业务模型，而是在已有模块上做增强，使项目具备更强的工程价值和面试表达价值。

## 2. 增强原则

- 每个技术点必须对应明确业务场景。
- 不为了堆技术而引入组件。
- 优先增强主链路中真实存在的问题。
- 每次增强都要能通过接口或日志体现效果。

## 3. 用户端增强

### 手机验证码登录

- 支持用户使用手机号和验证码登录。
- 验证码可先使用模拟发送，后续再接入短信服务。
- 验证码需要设置有效期。
- 验证码需要限制发送频率。
- 可使用 Redis 存储验证码。

### 预约修改与取消

- 用户可以在预约开始前取消预约。
- 用户可以在限定时间内修改预约时间或人数。
- 修改预约时需要重新校验桌位容量和桌位状态。
- 临近预约时间时，是否允许修改由商户参数控制。

### 预约超时提醒

- 用户预约即将超时时，可以收到系统提醒。
- V2 可先通过系统内记录或日志模拟通知。
- 后续可扩展为短信、站内信或 WebSocket 推送。

### 订单催单

- 用户可以对长时间未上齐的订单发起催单。
- 催单需要满足最小等待时间。
- 同一订单催单需要设置间隔限制。
- 可使用 Redis 做催单频率控制。

### 追加订单

- 用户在同一桌位用餐期间可以追加菜品。
- 追加订单可以独立生成子订单，也可以合并到同一用餐单。
- V2 建议先采用独立追加订单，降低复杂度。

## 4. 商户端增强

### 预约超时自动取消

- 店长可以配置预约超时时长。
- 定时任务定期扫描超时预约。
- 超时未到店的预约自动取消。
- 对应桌位状态恢复为空闲。
- 自动取消行为需要记录日志。
- 该能力只针对尚未到店的预约，不用于释放已经扫码占座的等待中桌位。

### 长时等待提醒

- 店长可以配置等待中桌位的提醒阈值，例如 20 分钟。
- 桌位进入等待中后，系统记录等待开始时间。
- 超过阈值仍未下单时，商户端显示长时等待提醒。
- 系统只提醒店员线下确认，不自动释放现实占用的桌位。
- 店员确认用户已离开后，可手动释放桌位。
- 店员确认用户仍在等待时，可继续保留等待中状态。

### 预约参数配置

- 店长可以配置可提前预约的最大时间范围。
- 店长可以配置预约保留时长。
- 店长可以配置是否启用自动取消。
- V2 可先作为单门店全局参数。

### 菜品批量导入导出

- 支持通过 csv 导入菜品。
- 导入菜品默认状态为停售。
- 支持导出当前菜品列表。
- 导入时需要校验名称、价格、分类、状态等字段。

### 菜品批量状态变更

- 支持批量启售、停售、设为售罄。
- 删除启售菜品前需要先停售，或执行停售并删除。
- 删除行为需要记录操作日志。

### 改单与退单

- 用户提出改单或退单请求。
- 店长审核后决定是否通过。
- 审核通过后修改订单或菜品项状态。
- V2 可以只支持店长操作，不做复杂审批流。

### 经营操作日志

- 记录桌位状态变化。
- 记录预约创建、确认、取消、超时取消。
- 记录订单状态变化。
- 记录菜品上下架。
- 记录员工关键操作。
- V2 可以先实现日志落库和按类型查询。

## 5. 员工与排班增强

### 员工信息完善

- 员工信息增加姓名、手机号、角色、账号状态、入职时间。
- 员工离职时账号改为禁用状态。
- 禁用员工不能登录商户端。

### 基础排班管理

- 店长可以为员工创建排班记录。
- 排班记录包括员工、日期、班次、开始时间、结束时间。
- 店员可以查看自己的排班。
- 店长可以查看全部排班。
- V2 暂不做复杂考勤统计。

### 班次管理

- 店长可以创建固定班次，如早班、晚班、全天班。
- 创建排班时可以选择已有班次。
- 班次只作为排班模板，不直接代表考勤结果。

## 6. 技术增强

### Redis 缓存

可优先用于以下场景：

- 缓存启售菜品列表。
- 缓存桌位状态列表。
- 存储短信验证码。
- 控制验证码发送频率。
- 控制催单频率。
- 防止重复提交。

### 防重复提交

- 对创建预约、创建订单等关键接口增加防重复提交。
- 可以使用请求幂等 Token 或 Redis 锁。
- V2 先选择一种简单可讲清楚的方案即可。

### 定时任务

可优先用于以下场景：

- 自动取消超时预约。
- 标记长时间未下单的等待中桌位并提醒商户端。
- 扫描长时间未处理订单。

### 消息队列

可优先用于以下场景：

- 订单创建后异步通知商户端。
- 预约即将超时提醒。
- 催单通知。
- 操作日志异步落库。

V2 中消息队列建议只选择一个明确场景落地，不要全面改造所有流程。

### Docker 化部署

- 编写 Dockerfile。
- 使用 docker-compose 启动 MySQL、Redis、后端服务。
- 如果引入 MQ，再加入 RabbitMQ 或 RocketMQ。
- 提供部署说明文档。

- V1 主流程保持稳定可运行。
- 至少一个 Redis 场景真实落地并可解释。
- 至少一个定时任务场景真实落地。
- 至少一个 MQ 或异步处理场景真实落地。
- Docker 本地部署流程可跑通。
- 关键业务操作有日志记录。
- README 能支撑别人理解和启动项目。
- 项目具备可写入简历的技术亮点。
