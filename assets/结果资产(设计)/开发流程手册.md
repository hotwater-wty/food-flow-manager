# 开发流程手册

## 1. 文档定位

本文档用于沉淀一套适合个人后端项目实践的开发流程。

它不是企业制度，也不是要求一次性全部掌握的理论材料，而是一份操作手册：当你启动一个新项目、梳理需求、进入设计、搭建工程或推进编码时，可以按阶段查看该做什么、产出什么、做到什么程度就可以进入下一步。

本文档的主要读者是未来的自己。

使用原则：

- 先保证项目能推进，再追求文档完整。
- 文档服务开发，不让文档反过来阻塞开发。
- 每个阶段只解决当前阶段最关键的问题。
- 开发时按需回查文档，不要求一次性记住所有字段、接口和细节。

## 2. 总体开发流程

一个后端项目从想法到编码，推荐按以下顺序推进：

```text
项目定位
-> 范围收敛
-> 需求分析
-> 业务流程设计
-> 领域模型设计
-> 核心表设计
-> 接口清单设计
-> 接口规范设计
-> 技术选型
-> 项目包结构设计
-> 数据库 SQL 初稿
-> 工程搭建
-> 编码迭代
-> 测试与修正
-> README / 简历化 / 面试材料整理
```

在“膳畅管家”项目中，实际经历过的主线是：

```text
项目定位 -> 范围收敛 -> 业务流程 -> 领域模型 -> 表设计 -> 接口设计 -> 接口规范 -> 技术选型 -> 包结构 -> SQL -> 工程搭建 -> 编码迭代
```

这条线的核心逻辑是：

```text
先想清楚业务
再想清楚数据
再想清楚接口
再准备工程
最后写代码
```

## 3. 阶段一：项目定位与范围收敛

### 阶段目标

明确项目为什么做、做给谁用、一期做到什么程度、哪些功能暂不做。

这一阶段的关键不是把功能想得越多越好，而是控制项目范围，避免项目一开始就失控。

### 输入材料

- 项目想法
- 学习目标或求职目标
- 目标用户和业务场景
- 自己当前技术水平

### 输出文档

- 项目计划书
- V1 最小业务闭环功能清单
- V2 业务增强功能清单

### 何时进入下一阶段

满足以下条件即可：

- 能用几句话说清项目定位。
- 能明确 V1 做什么。
- 能明确 V1 不做什么。
- 能说出 V2 主要增强方向。

不需要做到：

- 把所有未来功能都设计完整。
- 把数据库表和接口提前想清楚。
- 一开始就确定 Redis、MQ、支付、短信等所有技术细节。

### 常见风险

- 一期范围过大。
- 为了简历堆技术，导致项目无法完成。
- 把后续扩展功能提前塞进 V1。

### 当前项目案例

“膳畅管家”最初功能较多，后来拆成：

- V1：预约、到店、开台、点餐、订单处理、清台。
- V2：Redis、MQ、定时任务、催单、导入导出、排班、Docker 等增强能力。

对应文档：

- `餐饮预约与堂食排班管理系统项目计划书.md`
- `V1最小业务闭环功能清单.md`
- `V2业务增强功能清单.md`

## 4. 阶段二：需求分析

### 阶段目标

把“我要做什么功能”拆成角色、功能模块、主流程和边界条件。

需求分析阶段重点回答：

- 有哪些角色？
- 每个角色能做什么？
- 哪些能力属于 V1？
- 哪些能力暂不纳入？
- 哪些功能是主流程必须的？

### 输入材料

- 项目计划书
- V1/V2 功能清单
- 用户端和商户端场景描述

### 输出文档

- 角色与功能范围说明
- V1 功能清单
- V2 功能增强清单

如果项目不大，可以不单独创建“需求规格说明书”，而是用功能清单承接需求分析。

### 何时进入下一阶段

满足以下条件即可：

- 能说出系统角色。
- 能说出 V1 主业务链路。
- 能区分核心功能和增强功能。
- 能识别出高风险范围并推迟。

### 常见风险

- 把真实业务里的所有可能情况都塞进第一版。
- 没有明确角色边界。
- 没有定义“不做什么”。

### 当前项目案例

当前项目角色被收敛为：

- 普通用户
- 店员
- 店长

V1 不做：

- 在线支付
- 短信验证码
- Redis
- MQ
- 复杂排班
- 催单
- 改单/退单

对应文档：

- `V1最小业务闭环功能清单.md`
- `V2业务增强功能清单.md`

## 5. 阶段三：业务流程设计

### 阶段目标

把功能串成可运行的业务流程，明确状态如何变化、由谁触发、触发条件是什么。

这一阶段重点回答：

- 用户从哪里开始？
- 每一步由谁操作？
- 成功后状态怎么变？
- 失败时如何处理？
- 哪些状态不能自动变更？

### 输入材料

- V1 功能清单
- 角色划分
- 业务场景描述

### 输出文档

- 核心业务流程设计
- 状态流转说明

### 何时进入下一阶段

满足以下条件即可：

- 能说清主流程。
- 能说清关键状态。
- 能说清每个状态由谁触发变化。
- 能识别出需要人工确认的业务节点。

不需要做到：

- 把所有异常流程全部穷举。
- 把接口路径和表字段提前设计完。

### 常见风险

- 状态设计过多，导致实现复杂。
- 状态设计过少，无法表达真实业务。
- 系统替现实做了不该做的自动决策。

### 当前项目案例

“膳畅管家”的桌位状态设计为：

```text
空闲 -> 已预约：用户创建预约
已预约 -> 等待中：预约用户到店扫码核验通过
空闲 -> 等待中：未预约用户扫码占座成功
等待中 -> 用餐中：用户下单成功
等待中 -> 空闲：店员线下确认用户离开后释放
用餐中 -> 空闲：店员清台释放
```

后续又补充了：

```text
空闲 -> 禁用：商户禁用临时不可用桌位
禁用 -> 空闲：商户恢复桌位
```

对应文档：

- `核心业务流程设计.md`

## 6. 阶段四：领域模型与数据库设计

### 阶段目标

把业务流程中的核心对象识别出来，并进一步落成数据库表。

这一阶段重点回答：

- 业务里有哪些核心实体？
- 实体之间是什么关系？
- 哪些状态需要落库？
- 哪些字段是主流程必须的？
- 哪些表暂时不建？

### 输入材料

- 核心业务流程设计
- 状态流转说明
- V1 功能清单

### 输出文档

- 领域模型与核心表设计
- 核心状态枚举
- 主流程表结构校验
- SQL 初稿

### 何时进入下一阶段

满足以下条件即可：

- 能列出核心实体。
- 能说清实体关系。
- 能用表结构支撑主流程。
- 能通过主流程反推表结构没有明显缺口。

不需要做到：

- 一次性设计最终数据库。
- 给所有未来扩展功能建表。
- 过早追求完美索引和完整历史审计。

### 常见风险

- 只看到表，没看到业务对象。
- 表字段堆得太多。
- 缺少承接中间状态的实体。
- 把 V2 功能提前建表。

### 当前项目案例

当前项目中发现“等待中”状态不能只靠桌位表或订单表表达，因此引入了：

```text
DiningSession 开台会话
```

它用于记录一次真实到店占座过程，连接用户、桌位、预约和订单。

对应文档：

- `领域模型与核心表设计.md`
- `schema.sql`

## 7. 阶段五：接口设计与接口规范

### 阶段目标

明确系统能力如何对外提供，并统一接口格式。

接口设计阶段重点回答：

- 有哪些接口？
- 接口属于用户端还是商户端？
- 谁有权限访问？
- 请求需要哪些参数？
- 成功后影响哪些状态？
- 所有接口返回格式是否一致？

### 输入材料

- V1 功能清单
- 业务流程设计
- 领域模型与表设计

### 输出文档

- 核心接口清单设计
- 接口规范设计

### 何时进入下一阶段

满足以下条件即可：

- 主流程接口已经覆盖。
- 用户端和商户端接口边界清楚。
- 统一响应、分页、时间、字段命名、Token 传递方式已明确。
- 能通过接口反推主流程能跑通。

不需要做到：

- 手写所有接口的完整 JSON 示例。
- 在编码前把所有 DTO 字段完全定死。
- 用人工文档替代 Knife4j/OpenAPI 自动文档。

### 常见风险

- 接口路径混乱。
- 用户端和商户端职责混在一起。
- 前端传当前用户 ID，导致越权风险。
- 没有统一响应格式。

### 当前项目案例

当前项目约定：

```text
用户端接口：/api/user
商户端接口：/api/admin
Token：Authorization: Bearer xxx
统一响应：code / msg / data
```

下单接口围绕开台会话设计：

```text
POST /api/user/sessions/{sessionId}/orders
```

对应文档：

- `核心接口清单设计.md`
- `接口规范设计.md`

## 8. 阶段六：工程搭建前收口

### 阶段目标

把设计文档转成可以直接开始工程搭建的技术准备。

这一阶段重点回答：

- 用哪些技术？
- 暂不引入哪些技术？
- 包结构怎么设计？
- 基础类先写哪些？
- SQL 初稿是否能建库建表？

### 输入材料

- 技术目标
- 接口规范
- 表设计
- 项目复杂度

### 输出文档

- 技术选型与依赖规划
- 项目包结构设计
- 数据库 SQL 初稿

### 何时进入编码阶段

满足以下条件即可：

- 技术栈已确定。
- V1 暂不引入的技术已明确。
- 包结构已明确。
- SQL 初稿可执行。
- 工程可以创建并启动。

### 常见风险

- 过早引入 Redis、MQ、Spring Security 等复杂组件。
- 包结构不清，后续类随意放置。
- SQL 等到编码中后期才补，导致实现反复调整。

### 当前项目案例

当前项目 V1 技术选择：

```text
JDK 17
Spring Boot 3.x
Maven
MyBatis Plus
MySQL 8
JWT + 自定义拦截器
Knife4j / OpenAPI
Validation
Lombok
```

V1 暂不引入：

```text
Redis
MQ
Docker
Spring Security
短信服务
在线支付
```

对应文档：

- `技术选型与依赖规划.md`
- `项目包结构设计.md`
- `schema.sql`

## 9. 阶段七：编码阶段推进方式

### 阶段目标

按模块、按主流程逐步实现业务，避免同时展开过多模块。

推荐顺序：

```text
工程启动
-> 数据库连接
-> common 基础类
-> 用户注册登录
-> JWT 与拦截器
-> 桌位管理
-> 预约
-> 开台 session
-> 菜品
-> 订单
-> 主流程联调
```

### 输入材料

- 项目包结构设计
- 接口规范设计
- 核心接口清单设计
- schema.sql

### 输出内容

- 可运行代码
- Apifox 测试用例
- 开发排错笔记
- 设计文档必要更新

### 何时进入下一模块

满足以下条件即可：

- 当前模块能启动。
- 当前模块核心接口能用 Apifox 测通。
- 关键异常能返回统一格式。
- 代码结构符合包结构设计。

### 常见风险

- 未完成认证链路就开始写业务接口。
- 一次性写多个模块，难以定位错误。
- 遇到设计变化但不更新源文档。

### 当前项目案例

当前项目已完成：

- 工程搭建
- MyBatis Plus 配置
- 用户注册与登录基础逻辑
- JWT 生成、校验与拦截器
- 桌位模块开发中

当前阶段的经验记录方式：

- 具体报错写入 `开发排错笔记.md`
- 设计结论变化同步源文档
- 影响多个文档的设计变化写入 `设计文档更新日志.md`

## 10. 文档阅读方法

### 第一轮：只看目录

先看每份文档标题和小节标题，目标是知道它解决什么问题。

推荐顺序：

```text
项目计划书
V1 功能清单
核心业务流程设计
领域模型与核心表设计
核心接口清单设计
接口规范设计
技术选型与依赖规划
项目包结构设计
schema.sql
```

### 第二轮：只读当前阶段相关内容

编码时不要全量精读所有文档。

示例：

- 写登录：看接口规范、技术选型、包结构、用户认证接口。
- 写桌位：看桌位状态、桌位表、桌位接口。
- 写预约：看预约流程、预约状态、预约表、预约接口。
- 写订单：看开台会话、订单表、下单接口。

### 停止标准

读到以下程度即可停止：

- 能讲清 V1 主流程。
- 能说出核心对象。
- 能理解关键状态流转。
- 能知道接口大概按用户端和商户端分组。
- 能知道代码该放在哪些包。
- 能知道 V1 暂不做哪些技术增强。

不需要做到：

- 背出每个字段。
- 背出每个接口路径。
- 背出每个状态码。
- 一次性理解全部索引和所有边界条件。

## 11. 文档维护规则

当前项目采用以下维护规则：

- 设计结论变化：直接更新对应源文档。
- 有复盘价值的问题：写入 `设计与开发问题记录.md`。
- 具体报错、配置错误、工具使用问题：写入 `开发排错笔记.md`。
- 影响多个设计文档的变化：写入 `设计文档更新日志.md`。
- 小修、措辞调整、普通代码实现细节：不记录变更日志。

### 源文档优先

开发时以源文档为准，不以旧讨论记录为准。

例如桌位状态增加“禁用”后，应同步更新：

- 功能清单
- 领域模型与表设计
- 接口清单
- SQL
- 枚举代码

### 问题记录与排错笔记分离

`设计与开发问题记录.md` 用于记录设计取舍，例如：

- 为什么引入开台会话。
- 为什么下单接口围绕 session。
- 为什么 V1 不建物理外键。

`开发排错笔记.md` 用于记录具体错误，例如：

- Apifox Token 头传错导致 401。
- 注解 import 错误导致请求体无法绑定。
