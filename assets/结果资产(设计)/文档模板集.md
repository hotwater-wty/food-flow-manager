# 文档模板集

本文档提供后续 Java 后端项目可复用的设计文档模板。模板不是强制格式，可以按项目规模裁剪。

使用原则：

- 小项目可以合并文档。
- 中大型学习项目建议按模板逐步产出。
- 每份文档都要服务后续开发，不写无法指导实现的空话。
- 每个阶段只写到足够支撑下一阶段即可。

## 1. 项目计划书模板

### 文档用途

用于明确项目为什么做、做什么、有什么价值、后续如何推进。

### 适用阶段

项目启动前。

### 推荐结构

```md
# 项目计划书

## 1. 项目定位
## 2. 项目背景
## 3. 项目目标
### 业务目标
### 技术目标
### 个人成长 / 求职目标
## 4. 项目范围
### 一期核心范围
### 暂不纳入一期范围
## 5. 项目价值
## 6. 技术规划
## 7. 阶段规划
## 8. 风险与控制
## 9. 预期成果
```

### 填写要求

- 项目定位要具体，不写泛泛的“管理系统”。
- 一期范围必须明确。
- 暂不做的内容必须写出来。
- 技术规划只写大方向，不提前堆依赖。

### 停止标准

能回答：

- 为什么做这个项目？
- 给谁用？
- 一期要做到什么程度？
- 哪些功能暂时不做？

### 示例片段

```md
本项目面向中小型餐饮商户，聚焦预约、堂食点餐、桌位管理和订单处理，不追求覆盖完整餐饮 SaaS。
```

## 2. V1 最小业务闭环功能清单模板

### 文档用途

用于确定第一版必须完成的功能范围。

### 适用阶段

需求分析和范围收敛阶段。

### 推荐结构

```md
# V1 最小业务闭环功能清单

## 1. 版本目标
## 2. 用户角色
## 3. 用户端功能
## 4. 商户端 / 管理端功能
## 5. 系统基础能力
## 6. V1 不包含的内容
## 7. V1 验收标准
```

### 填写要求

- 每个功能只写 V1 必需内容。
- 不确定的增强功能放到 V2。
- 验收标准必须能通过接口或运行效果验证。

### 停止标准

能说出 V1 主流程，并且没有明显的范围失控。

### 示例片段

```md
V1 目标是打通“用户预约 -> 到店开台 -> 点餐下单 -> 商户处理 -> 清台释放”的主流程。
```

## 3. V2 业务增强功能清单模板

### 文档用途

用于承接暂不进入 V1 的增强能力，避免丢失想法，也避免干扰 V1。

### 适用阶段

V1 范围收敛后。

### 推荐结构

```md
# V2 业务增强功能清单

## 1. 版本目标
## 2. 增强原则
## 3. 用户端增强
## 4. 商户端 / 管理端增强
## 5. 技术增强
## 6. 文档与部署增强
## 7. V2 验收标准
```

### 填写要求

- 每个增强点要说明业务价值。
- 技术增强必须对应真实业务场景。
- 不把 V2 内容反向塞回 V1。

### 停止标准

V1 边界变清晰，增强想法有地方存放。

### 示例片段

```md
Redis 可用于验证码、热点菜品缓存、防重复提交和催单频率控制。
```

## 4. 核心业务流程设计模板

### 文档用途

用于描述主业务如何流转，以及核心状态如何变化。

### 适用阶段

需求范围明确后、表设计前。

### 推荐结构

```md
# 核心业务流程设计

## 1. 主流程概览
## 2. 流程一：xxx
## 3. 流程二：xxx
## 4. 流程三：xxx
## 5. 核心状态流转
## 6. 人工处理节点
```

### 填写要求

- 每个流程写清触发人、触发条件、结果状态。
- 状态流转要明确“谁触发”。
- 不让系统自动处理无法判断的现实情况。

### 停止标准

能从头到尾讲清主流程和关键状态变化。

### 示例片段

```md
等待中桌位代表用户已实际占座但尚未下单。系统可以提醒商户端，但不自动释放桌位。
```

## 5. 领域模型与核心表设计模板

### 文档用途

用于识别核心业务对象，并初步设计数据库表。

### 适用阶段

业务流程设计后、接口设计前。

### 推荐结构

```md
# 领域模型与核心表设计

## 1. 设计目标
## 2. 核心领域对象
## 3. 实体关系
## 4. 核心状态枚举
## 5. 核心表初稿
## 6. V1 暂不建表的内容
## 7. 主流程表结构校验
```

### 填写要求

- 先识别业务对象，再设计表。
- 每张表只放当前主流程需要的字段。
- 状态字段要和业务流程一致。
- 明确哪些表暂不建。

### 停止标准

能用表结构支撑主流程，不存在明显数据缺口。

### 示例片段

```md
开台会话用于记录一次真实到店占座过程，连接用户、桌位、预约和订单。
```

## 6. 核心接口清单设计模板

### 文档用途

用于确定系统对外暴露哪些接口，以及接口的权限、参数和业务规则。

### 适用阶段

核心表设计后、编码前。

### 推荐结构

```md
# 核心接口清单设计

## 1. 设计目标
## 2. 接口设计约定
## 3. 核心接口列举
## 4. V1 主流程接口设计
## 5. 管理端基础接口清单
## 6. 当前暂不设计的接口
```

单个接口推荐格式：

```md
### 接口名称

- 请求方式：
- 请求路径：
- 权限：
- 请求参数：
- 业务规则：
- 响应数据：
```

### 填写要求

- 路径体现端侧和资源归属。
- 不让前端传当前登录用户 ID。
- 主流程接口要写业务规则。
- 增强接口可以只列清单，后续再细化。

### 停止标准

主流程接口完整，权限边界清楚。

### 示例片段

```md
POST /api/user/sessions/{sessionId}/orders
```

## 7. 接口规范设计模板

### 文档用途

用于统一所有接口的返回格式、分页格式、时间格式、Token 传递方式等。

### 适用阶段

接口清单设计后、编码前。

### 推荐结构

```md
# 接口规范设计

## 1. 文档目标
## 2. 统一响应格式
## 3. 分页响应格式
## 4. 错误码规则
## 5. 时间格式
## 6. 字段命名规则
## 7. Token 传递方式
## 8. HTTP 方法约定
## 9. 参数校验规则
## 10. 金额格式
## 11. 空值与空列表
## 12. 后续落地方式
```

### 填写要求

- 响应结构必须统一。
- 时间格式必须明确。
- Token 传递方式必须和拦截器实现一致。
- 不要手写所有接口完整 JSON，后续交给 OpenAPI/Knife4j。

### 停止标准

能指导 `Result`、`PageResult`、异常处理器、拦截器的实现。

### 示例片段

```json
{
  "code": 1,
  "msg": "success",
  "data": {}
}
```

## 8. 技术选型与依赖规划模板

### 文档用途

用于明确项目采用哪些技术，以及哪些技术暂不引入。

### 适用阶段

工程搭建前。

### 推荐结构

```md
# 技术选型与依赖规划

## 1. 文档目标
## 2. 项目基础信息
## 3. 核心技术选型
## 4. V1 建议依赖清单
## 5. V1 暂不引入的技术
## 6. 推荐项目配置文件
## 7. 编码阶段优先落地的技术能力
## 8. 技术风险与控制
## 9. 当前结论
```

### 填写要求

- 写明具体版本倾向。
- 写明选择原因。
- 写明暂不引入的原因。
- 不为简历堆技术。

### 停止标准

能据此创建 Spring Boot 工程和 `pom.xml`。

### 示例片段

```md
V1 使用 JWT + 自定义拦截器，暂不引入 Spring Security。
```

## 9. 项目包结构设计模板

### 文档用途

用于规定代码放置位置，避免工程结构混乱。

### 适用阶段

工程搭建前。

### 推荐结构

```md
# 项目包结构设计

## 1. 设计目标
## 2. 基础包名
## 3. 推荐目录结构
## 4. 公共层说明
## 5. 配置与拦截层说明
## 6. 业务模块说明
## 7. 模块内部结构说明
## 8. 各模块职责建议
## 9. 命名建议
## 10. 资源目录建议
## 11. 优先实现的基础类
## 12. 注意事项
```

### 填写要求

- 明确按业务模块分包还是按技术层分包。
- 明确 Entity、DTO、VO 区别。
- 明确 Controller 不直接调用 Mapper。
- 明确复杂状态变更放 Service 层。

### 停止标准

能据此创建包结构和第一批基础类。

### 示例片段

```text
module/user
├── controller
├── dto
├── entity
├── mapper
├── service
└── vo
```

## 10. 数据库 SQL 初稿模板

### 文档用途

用于把核心表设计转成可执行 SQL。

### 适用阶段

核心表设计后、工程连接数据库前。

### 推荐结构

```sql
CREATE DATABASE IF NOT EXISTS `project_db`;
USE `project_db`;

CREATE TABLE IF NOT EXISTS `xxx` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  ...
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='xxx表';
```

### 填写要求

- 表名和字段名使用下划线。
- 主键统一。
- 核心编号加唯一索引。
- 查询字段加必要索引。
- 金额用整数分。
- 时间字段统一。

### 停止标准

SQL 能执行建库建表，支撑 V1 主流程启动开发。

### 示例片段

```sql
`status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0空闲，1已预约'
```

## 11. 设计与开发问题记录模板

### 文档用途

用于记录有复盘价值的设计取舍，后续可整理为 README、简历、面试素材。

### 适用阶段

设计阶段和开发阶段持续维护。

### 推荐结构

```md
# 设计与开发问题记录

## 1. 问题标题

### 问题背景
### 方案思考
### 当前结论
### 面试表达素材
```

### 填写要求

- 只记录有设计价值的问题。
- 不记录普通语法错误。
- 写清为什么不是另一个方案。

### 停止标准

未来能通过该记录复盘当时的设计取舍。

### 示例片段

```md
下单接口围绕开台会话设计，而不是单纯围绕用户或桌位。
```

## 12. 开发排错笔记模板

### 文档用途

用于记录具体开发错误、配置错误、工具使用问题，定位为个人错题本。

### 适用阶段

编码阶段持续维护。

### 推荐结构

```md
# 开发排错笔记

## 1. 问题标题

### 现象
### 错误写法
### 原因
### 正确写法
### 经验
```

### 填写要求

- 记录真实错误现象。
- 写出错误代码或错误配置。
- 写出正确写法。
- 总结排查顺序。

### 停止标准

下次遇到同类错误时能快速定位。

### 示例片段

```md
Apifox 中应使用 Authorization: Bearer <jwt>，而不是 token: <jwt>。
```

## 13. 设计文档更新日志模板

### 文档用途

用于记录影响多个设计文档的设计结论变化，并标明去哪里查最新内容。

### 适用阶段

设计和开发阶段持续维护。

### 推荐结构

```md
# 设计文档更新日志

## yyyy-MM-dd

### 变更标题

变更内容：

- ...

查看位置：

- ...
```

### 填写要求

- 只记录设计结论变化。
- 不记录普通措辞修改。
- 不代替源文档。
- 源文档必须同步更新。

### 停止标准

未来能通过更新日志知道哪些设计发生过重要变化，以及应该去哪些文档查看。

### 示例片段

```md
桌位状态增加“禁用”，查看位置：V1 功能清单、领域模型、接口清单、schema.sql、TableStatusEnum。
```
