# 领域模型与核心表设计

## 1. 设计目标

本文档用于把 V1 的核心业务流程落到领域对象和数据库表初稿上。

V1 重点支撑以下主流程：

> 用户预约桌位 -> 到店扫码开台 -> 堂食下单 -> 商户处理订单 -> 用餐完成 -> 店员清台释放桌位

当前阶段先保证表结构能支撑主流程，不提前引入短信、支付、催单、排班、复杂日志、复杂权限配置等 V2 能力。

## 2. 核心领域对象

### 用户 User

说明：到店用餐的普通用户，可以预约桌位、扫码开台、创建堂食订单、查看个人记录。

### 员工 Employee

说明：商户端工作人员，可以处理桌位、预约、菜品和订单。V1 暂分为店员和店长两类角色。

### 桌位 DiningTable

说明：餐厅内可预约、可扫码占座、可用餐的桌位。桌位状态用于表示当前经营占用情况。

### 预约 Reservation

说明：用户在线上提前预订的到店时间、人数和桌位信息。预约属于未来资源占用。

### 开台 DiningSession

说明：用户到店扫码后产生的一次桌位占用会话，用于记录“等待中”和“用餐中”的现实占座过程。

引入该对象的原因是：桌位进入等待中后，用户可能还没有下单，此时不能只依赖订单表表达占座关系，否则系统无法知道是谁占用了桌位。

### 菜品分类 DishCategory

说明：菜品的基础分类，如热菜、凉菜、主食、饮品。V1 可做简单分类管理，也可以先预置固定分类。

### 菜品 Dish

说明：用户可查看并选择的商品。用户端只展示启售状态的菜品。

### 订单 DiningOrder

说明：用户在某次开台会话中提交的堂食订单。为了避免使用数据库关键字，不建议直接命名为 `order` 表。

### 订单明细 OrderItem

说明：订单中的具体菜品项，记录菜品、数量、下单时价格等信息。

## 3. 实体关系

- 一个用户可以创建多个预约。
- 一个预约只能关联一个桌位。
- 一个用户可以发起多次开台。
- 一个开台会话只能关联一个桌位。
- 一个开台会话可以来自一个预约，也可以来自未预约扫码占座。
- 一个开台会话可以产生一个或多个订单。V1 建议先按一个会话一个主订单实现，V2 再扩展追加订单。
- 一个订单包含多个订单明细。
- 一个订单明细对应一个菜品。
- 一个菜品属于一个菜品分类。
- 一个员工可以处理多个订单状态变更。
- 一个员工可以手动释放桌位或维护桌位状态。

## 4. 核心状态枚举

### 用户状态 user.status

- 1：正常
- 2：禁用

V1 不需要设计过多用户状态。

### 员工状态 employee.status

- 1：正常
- 2：禁用
- 3：离职

“下岗”“封禁中”等概念暂不进入 V1，避免员工管理变成人事系统。

### 员工角色 employee.role

- 1：店员
- 2：店长

V1 只做角色区分，不做动态权限配置。

### 桌位状态 dining_table.status

- 0：空闲
- 1：已预约
- 2：等待中
- 3：用餐中
- 4：禁用

说明：

- 已预约表示未来预约占用。
- 等待中表示用户已经到店并扫码占座，但尚未下单。
- 用餐中表示用户已经下单，正在用餐或等待上菜。
- 禁用表示桌位临时不可用，例如维修、撤桌、包间暂停使用。
- 等待中桌位不自动释放，只能由店员线下确认后手动释放。
- 用餐中桌位不自动释放，只能由店员线下清台后手动释放。
- 禁用桌位不允许预约和扫码占座。

### 预约状态 reservation.status

- 0：待到店
- 1：已到店
- 2：已取消
- 3：已超时

V1 可先实现待到店、已到店、已取消；预约超时可先人工处理，V2 再通过定时任务自动处理。

### 开台状态 dining_session.status

- 0：等待中
- 1：用餐中
- 2：已完成
- 3：已取消

说明：

- 等待中：扫码成功，占座但未下单。
- 用餐中：已创建订单。
- 已完成：用餐结束，店员清台。
- 已取消：用户未下单离开，店员确认后释放。

### 菜品状态 dish.status

- 0：停售
- 1：启售
- 2：售罄

v1的售罄状态只做预留，v2再通过其实现菜品限量销售。

### 订单状态 dining_order.status

- 1：已下单
- 2：制作中
- 3：已上齐
- 4：已完成
- 5：已取消

## 5. 核心表初稿

### user 用户表

- id：主键
- phone：手机号，唯一
- password：加密后的密码
- nickname：昵称
- status：用户状态，1 正常，2 禁用
- create_time：创建时间
- update_time：更新时间

### employee 员工表

- id：主键
- phone：手机号，唯一
- password：加密后的密码
- name：员工姓名
- role：员工角色，1 店员，2 店长
- status：员工状态，1 正常，2 禁用，3 离职
- create_time：创建时间
- update_time：更新时间

### dining_table 桌位表

- id：主键
- table_no：桌号，唯一
- capacity：建议容纳人数
- location_desc：位置描述
- status：桌位状态，0 空闲，1 已预约，2 等待中，3 用餐中，4 禁用
- current_session_id：当前开台会话 ID，可为空
- create_time：创建时间
- update_time：更新时间

说明：

- `current_session_id` 用于快速定位当前占用该桌位的开台会话。
- 如果桌位为空闲、已预约或禁用，该字段可以为空。

### reservation 预约表

- id：主键
- reservation_no：预约编号，唯一
- user_id：预约用户 ID
- table_id：预约桌位 ID
- people_count：预约人数
- reserve_time：预约到店时间
- status：预约状态，0 待到店，1 已到店，2 已取消，3 已超时
- check_in_time：实际到店时间，可为空
- cancel_time：取消时间，可为空
- create_time：创建时间
- update_time：更新时间

说明：

- 原 `book_time` 建议改为 `reserve_time`，语义更接近“预约到店时间”。
- 预约成功后，桌位状态变为已预约。
- 预约用户到店扫码核验成功后，预约状态变为已到店，并创建开台会话。

### dining_session 开台表

- id：主键
- session_no：开台编号，唯一
- user_id：用户 ID
- table_id：桌位 ID
- reservation_id：预约 ID，可为空
- status：开台状态，0 等待中，1 用餐中，2 已完成，3 已取消
- open_time：扫码开台时间
- first_order_time：首次下单时间，可为空
- close_time：结束时间，可为空
- close_employee_id：清台或释放桌位的员工 ID，可为空
- create_time：创建时间
- update_time：更新时间

说明：

- 预约用户到店扫码，会创建一条开台记录，并关联 `reservation_id`。
- 未预约用户扫码占座，也会创建一条开台记录，但 `reservation_id` 为空。
- 等待中长时间未下单时，不自动释放桌位；店员确认用户离开后，将开台状态改为已取消，并释放桌位。

### dish_category 菜品分类表

- id：主键
- name：分类名称
- sort：排序值
- status：状态，1 启用，2 禁用
- create_time：创建时间
- update_time：更新时间

### dish 菜品表

- id：主键
- category_id：菜品分类 ID
- name：菜品名称
- price：价格，建议使用分作为整数存储
- image：配图地址
- description：描述信息
- status：菜品状态，0 停售，1 启售，2 售罄
- create_time：创建时间
- update_time：更新时间

说明：

- 用户端只展示启售状态的菜品。
- 价格建议用整数分存储，避免浮点精度问题。

### dining_order 订单表

- id：主键
- order_no：订单编号，唯一
- user_id：用户 ID
- table_id：桌位 ID
- session_id：开台会话 ID
- total_amount：订单总金额，单位分
- status：订单状态，1 已下单，2 制作中，3 已上齐，4 已完成，5 已取消
- create_time：创建时间
- update_time：更新时间

说明：

- 不建议把表名直接命名为 `order`，因为它容易和 SQL 关键字冲突。
- V1 不接入真实支付，订单创建成功即视为已下单。
- 用户首次下单成功后，开台状态从等待中变为用餐中，桌位状态从等待中变为用餐中。

### order_item 订单明细表

- id：主键
- order_id：订单 ID
- dish_id：菜品 ID
- dish_name：下单时菜品名称
- dish_image：下单时菜品图片
- dish_price：下单时菜品单价，单位分
- quantity：数量
- amount：该明细总金额，单位分
- remark：口味或备注，可为空
- create_time：创建时间
- update_time：更新时间

说明：

- 菜品名称、图片、价格需要在订单明细中冗余保存。
- 原因是后续菜品改名、改价或删除时，历史订单仍需要展示当时的下单信息。

## 6. V1 暂不建表的内容

以下内容暂不进入 V1 核心表设计：

- 短信验证码表：V2 可使用 Redis 存储。
- 支付流水表：V1 不接入真实支付。
- 催单表：V2 再设计。
- 退单/改单申请表：V2 再设计。
- 员工排班表：V2 再设计。
- 操作日志表：V1 先使用应用日志，V2 再考虑落库查询。
- 动态权限表：V1 只使用固定角色。

## 7. 主流程表结构校验

### 用户创建预约

- 写入 `reservation`。
- 校验桌位状态是否为空闲。
- 更新 `dining_table.status` 为已预约。

### 预约用户到店扫码

- 校验 `reservation` 是否属于当前用户、桌位是否匹配、预约是否有效。
- 更新 `reservation.status` 为已到店。
- 创建 `dining_session`，状态为等待中。
- 更新 `dining_table.status` 为等待中，写入 `current_session_id`。

### 未预约用户扫码占座

- 校验桌位状态是否为空闲。
- 创建 `dining_session`，状态为等待中。
- 更新 `dining_table.status` 为等待中，写入 `current_session_id`。

### 商户禁用桌位

- 校验桌位状态是否为空闲。
- 更新 `dining_table.status` 为禁用。
- 禁用桌位不允许预约或扫码占座。

### 商户恢复禁用桌位

- 校验桌位状态是否为禁用。
- 更新 `dining_table.status` 为空闲。

### 用户创建堂食订单

- 校验 `dining_session` 是否属于当前用户，且状态为等待中。
- 写入 `dining_order`。
- 写入多条 `order_item`。
- 更新 `dining_session.status` 为用餐中，并记录 `first_order_time`。
- 更新 `dining_table.status` 为用餐中。

### 商户处理订单

- 店员或店长更新 `dining_order.status`。
- V1 只更新订单当前状态，不单独记录状态流转表。

### 店员清台释放桌位

- 更新 `dining_order.status` 为已完成。
- 更新 `dining_session.status` 为已完成，记录 `close_time` 和 `close_employee_id`。
- 更新 `dining_table.status` 为空闲，清空 `current_session_id`。

### 店员释放未下单的等待中桌位

- 店员线下确认用户已离开。
- 更新 `dining_session.status` 为已取消，记录 `close_time` 和 `close_employee_id`。
- 更新 `dining_table.status` 为空闲，清空 `current_session_id`。
