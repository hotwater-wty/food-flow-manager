# V1阶段开发计划

现在应进入 **V1 主业务闭环开发**。后续不要随意扩功能，严格围绕这条链路推进：

```text
用户预约桌位 -> 到店扫码开台 -> 查看菜品 -> 创建堂食订单 -> 商户处理订单 -> 店员清台释放桌位
```

**后续开发顺序**

1. **预约模块 Reservation**

目标：让用户能预约空闲桌位。

需要完成：

- `reservation` 实体、Mapper、Service、Controller
- 用户创建预约：`POST /api/user/reservations`
- 用户查看我的预约：`GET /api/user/reservations`
- 用户取消预约：`POST /api/user/reservations/{reservationId}/cancel`
- 管理端查看预约列表 / 详情 / 取消异常预约

重点规则：

- 只能预约 `FREE` 桌位
- 禁用桌位不能预约
- 预约人数不能超过桌位容量
- 创建预约成功后：`dining_table.status = RESERVED`
- 取消预约后：`reservation.status = CANCELED`，桌位恢复 `FREE`

这一模块要重点练习：**事务控制**。创建预约时，写预约表和改桌位状态必须在同一个事务里。

2. **开台会话模块 DiningSession**

目标：让“扫码占座”这件事有数据承载。

需要完成：

- `dining_session` 实体、Mapper、Service、Controller
- 预约用户扫码到店：`POST /api/user/reservations/{reservationId}/check-in`
- 未预约用户扫码占座：`POST /api/user/tables/{tableId}/sessions`
- 用户查看当前开台：`GET /api/user/sessions/current`
- 店员释放等待中桌位：`POST /api/admin/sessions/{sessionId}/cancel`

重点规则：

- 预约扫码：校验预约属于当前用户、预约桌位和扫码桌位一致
- 未预约扫码：桌位必须是 `FREE`
- 成功后创建 `dining_session`
- 桌位进入 `WAITING`
- `dining_table.current_session_id` 写入当前开台 id
- 等待中不自动释放，只能店员手动释放

这一模块是整个项目的核心之一，因为它解决了“用户已占座但未下单”的业务表达问题。

3. **菜品分类与菜品模块 Dish / DishCategory**

目标：让用户可以点菜，让商户可以维护菜品。

建议先做简单版：

- 分类可以先只做查询，使用 SQL 初始化数据
- 菜品管理做 CRUD
- 用户端只看启售菜品

接口：

- 用户查看启售菜品：`GET /api/user/dishes`
- 用户查看菜品详情：`GET /api/user/dishes/{dishId}`
- 管理端新增 / 修改 / 删除 / 修改状态 / 查询菜品

重点规则：

- 菜品价格用“分”存储，使用 `Integer`
- 用户端只能查 `ON_SALE`
- 已下单历史不能因为菜品改价受到影响，所以订单明细后面要冗余保存菜名、价格

4. **堂食订单模块 DiningOrder / OrderItem**

目标：打通用户从等待中到用餐中的核心流程。

需要完成：

- `dining_order` 实体、Mapper、Service
- `order_item` 实体、Mapper、Service
- 用户创建堂食订单：`POST /api/user/sessions/{sessionId}/orders`
- 用户查看我的订单列表 / 详情
- 管理端查看订单列表 / 详情
- 管理端更新订单状态

重点规则：

- 当前用户必须拥有这个 `sessionId`
- 开台状态必须是 `WAITING`
- 桌位状态必须是 `WAITING`
- 菜品必须存在且启售
- 数量必须大于 0
- 计算订单总价
- 创建订单明细时冗余菜品名称、图片、价格
- 下单成功后：
  - `dining_session.status = DINING`
  - `dining_session.first_order_time = now`
  - `dining_table.status = DINING`

这一模块也必须使用事务。

5. **清台释放模块**

目标：完成 V1 最后一段业务闭环。

接口：

- `POST /api/admin/sessions/{sessionId}/close`

重点规则：

- 开台必须是 `DINING`
- 关联订单更新为 `COMPLETED`
- 开台更新为 `COMPLETED`
- 桌位恢复 `FREE`
- 清空 `current_session_id`

做到这里，V1 主链路基本跑通。

6. **员工模块与管理端权限，暂挂但最终必须补**

你现在说先排除管理端权限问题，可以暂时不做。但它不能永久跳过。

后面至少要补：

- 员工登录：`POST /api/admin/auth/login`
- token 中区分 `loginType`
- token 中携带员工 `role`
- 普通用户 token 不能访问 `/api/admin/**`
- 店员不能访问店长专属接口，例如删除桌位、管理员工

建议等预约、开台、菜品、订单主流程完成后，再统一补权限，否则现在会打断业务开发节奏。

**推荐当前下一步**

下一步直接做：**预约模块 Reservation**。

你可以按这个小任务拆：

1. 建 `Reservation` 实体、Mapper。
2. 建预约状态枚举 `ReservationStatusEnum`。
3. 建 `ReservationCreateDTO`、`ReservationVO`。
4. 写 `createReservation`，先只实现创建预约。
5. 用 `@Transactional` 包住创建预约和更新桌位状态。
6. 用 Apifox 测试：
   - 空闲桌位预约成功
   - 禁用桌位预约失败
   - 已预约桌位再次预约失败
   - 人数超过容量失败
   - 成功后桌位状态变成已预约

做到这里再继续“查看我的预约”和“取消预约”。这一步最适合作为你下一轮练习点。
