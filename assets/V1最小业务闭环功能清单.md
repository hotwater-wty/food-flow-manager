# V1 最小业务闭环功能清单

## 1. 版本目标

V1 的目标不是做完整餐饮管理系统，而是先打通一条可运行、可测试、可讲解的核心业务链路：

> 用户预约桌位 -> 到店开台 -> 堂食点餐 -> 商户处理订单 -> 结束用餐并释放桌位

本版本优先保证业务闭环、接口清晰、状态流转合理。暂不追求短信、支付、消息队列、复杂排班、导入导出等增强能力。

## 2. 用户角色

### 普通用户

面向到店用餐的顾客，主要负责预约、查看桌位、创建堂食订单、查看个人记录。

### 店员

面向商户日常操作人员，主要负责桌位状态维护、预约确认、订单处理、菜品基础维护。

### 店长

拥有商户端最高权限，除店员能力外，还可以管理员工账号和系统基础参数。

V1 暂定只区分普通用户、店员、店长三类角色，不做灵活的权限配置系统。

## 3. 用户端功能

### 用户注册

- 用户可以使用手机号和密码注册。
- 手机号需要唯一。
- 密码需要加密存储。
- V1 暂不接入短信验证码。

### 用户登录

- 用户使用手机号和密码登录。
- 登录成功后返回 JWT。
- 后续访问用户端受保护接口时，需要携带 Token。

### 查看桌位

- 用户可以查看当前桌位列表。
- 展示桌位号、座位数、位置描述、桌位状态。
- 桌位状态包括：空闲、已预约、等待中、用餐中、禁用。
- V1 只展示当前状态，不展示复杂的预计等待时间。
- 等待中表示用户已到店并扫码占座，但尚未下单。
- 禁用表示桌位临时不可用，不允许预约或扫码占座。

### 创建预约

- 已登录用户可以创建预约。
- 创建预约时需要选择桌位、预约人数、预约到店时间。
- 只能预约空闲桌位。
- 预约人数不能超过桌位建议容纳人数。
- 创建成功后，桌位状态变为已预约，返回用户预约编号。

### 查看我的预约

- 用户可以查看自己的预约记录。
- 展示预约编号、桌位信息、预约人数、预约时间、预约状态。
- V1 支持查看，不支持用户主动修改预约。
- 如需修改预约，后续版本再扩展。

### 创建堂食订单

- 已登录用户可以创建堂食订单。
- 下单时需要关联桌位。
- 下单内容包括菜品 ID、数量、备注。
- V1 暂不实现购物车，前端或接口调用方直接提交菜品列表。
- V1 不接入真实支付，订单创建成功即视为已下单。

### 查看我的订单

- 用户可以查看自己的订单记录。
- 展示订单编号、桌位、菜品明细、订单金额、订单状态、创建时间。
- V1 不支持催单、改单、退单。

## 4. 商户端功能

### 员工登录

- 店员和店长使用手机号和密码登录。
- 登录成功后返回 JWT。
- 根据角色判断可访问的商户端接口。

### 管理桌位

- 店员和店长可以新增、修改、查看桌位。
- 店长可以删除桌位。
- 桌位字段包括桌位号、座位数、位置描述、状态。
- 桌位状态包括：空闲、已预约、等待中、用餐中、禁用。
- 店员和店长可以将空闲桌位禁用，或将禁用桌位恢复为空闲。
- 禁用桌位不允许预约和扫码占座。

### 桌位状态维护

- 店员和店长可以根据实际经营情况更新桌位状态。
- 用户预约成功后，系统自动将桌位改为已预约。
- 用户到店扫码核验通过后，系统自动将桌位从已预约改为等待中。
- 用户未预约直接到店时，扫码占座成功后，系统自动将空闲桌位改为等待中。
- 等待中桌位在用户下单成功后，系统自动改为用餐中。
- 等待中桌位长时间未下单时，系统不自动释放桌位，只由店员线下确认后手动释放。
- 用户用餐结束并清台后，店员可将桌位改为空闲。
- 只有空闲桌位允许被禁用。

### 管理预约

- 店员和店长可以查看全部预约记录。
- 店员和店长可以确认用户到店。
- 店员和店长可以取消异常预约。
- V1 暂不做预约超时自动取消，只允许人工处理。

### 管理菜品

- 店员和店长可以新增、修改、查看菜品。
- 店长可以删除菜品。
- 菜品字段包括名称、价格、分类、描述、状态。
- 菜品状态包括：启售、停售、售罄。
- 用户只能看到启售状态的菜品。

### 处理堂食订单

- 店员和店长可以查看堂食订单列表。
- 店员和店长可以查看订单详情。
- 店员和店长可以更新订单状态。
- V1 订单状态建议包括：已下单、制作中、已上齐、已完成、已取消。

### 管理员工

- 店长可以新增员工账号。
- 店长可以启用或禁用员工账号。
- 店长可以查看员工列表。
- V1 只区分店员和店长，不做复杂职位体系。
- V1 不记录工资额度。

## 5. 系统基础能力

### 登录鉴权

- 用户端和商户端都使用 JWT 鉴权。
- 登录接口不需要 Token。
- 受保护接口需要校验 Token。

### 权限控制

- 普通用户只能访问用户端接口。
- 店员可以访问大部分商户端经营接口。
- 店长可以访问全部商户端接口。
- 禁用状态的员工不能登录。

### 统一返回结果

- 所有接口使用统一响应结构。
- 成功响应和失败响应格式保持一致。

### 统一异常处理

- 使用全局异常处理器处理业务异常和系统异常。
- 业务异常返回明确错误信息。
- 系统异常返回通用错误信息，并记录日志。

### 接口文档

- 项目内集成 Knife4j 或 OpenAPI 文档。
- 日常接口调试可以使用 Apifox。
- V1 每个接口需要有基本说明、请求参数、响应示例。

### 日志记录

- V1 先使用普通应用日志记录关键操作。
- 重点记录登录失败、预约创建、桌位状态变化、订单状态变化。
- 暂不设计复杂的业务日志查询系统。

## 6. V1 不包含的内容

- 短信验证码登录
- 阿里云短信服务
- 真实在线支付
- 购物车
- 催单
- 改单和退单流程
- 预约超时自动取消
- MQ 异步通知
- Redis 缓存
- 菜品 csv 导入导出
- 员工工资管理
- 员工排班管理
- 多门店管理
- 前端页面

## 7. V1 验收标准

- 用户可以注册、登录并获取 Token。
- 店员和店长可以登录并访问商户端接口。
- 用户可以查看桌位并创建预约。
- 预约创建后，桌位状态能正确变化。
- 用户扫码到店或扫码占座后，桌位能进入等待中。
- 用户下单成功后，桌位能进入用餐中。
- 店员可以手动释放等待中或用餐中的桌位。
- 用户可以查看启售菜品并创建堂食订单。
- 商户可以查看订单并更新订单状态。
- 用户可以查看自己的预约和订单记录。
- 接口文档可以正常访问。
- 项目可以在本地完整启动并通过接口测试工具跑通主流程。
