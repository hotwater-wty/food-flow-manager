# 核心接口清单设计

## 1. 设计目标

本文档用于设计 V1 阶段的核心接口，重点支撑以下主流程：

> 用户注册登录 -> 查看桌位 -> 创建预约 -> 到店扫码开台 -> 查看菜品 -> 创建堂食订单 -> 商户处理订单 -> 店员清台释放桌位

V1 只设计主流程必需接口和基础管理接口，不展开短信、支付、催单、排班、导入导出、复杂日志等 V2 能力。

## 2. 接口设计约定

- 用户端接口统一使用 `/api/user` 前缀。
- 商户端接口统一使用 `/api/admin` 前缀。
- 登录、注册接口不需要 Token。
- 其他受保护接口通过请求头携带 Token，文档中不再把 `token` 和当前用户 `id` 写成业务参数。
- 当前登录用户 ID 由后端从 Token 中解析，不由前端直接传入。
- 创建订单等依赖具体业务上下文的接口，优先围绕 `dining_session` 设计。

## 4. V1 主流程接口设计

### 4.1 用户注册

- 请求方式：POST
- 请求路径：`/api/user/auth/register`
- 权限：无需登录
- 请求参数：
  - `phone`：手机号
  - `password`：密码
  - `nickname`：昵称，可选
- 业务规则：
  - 手机号不能为空。
  - 手机号需要唯一。
  - 密码需要加密后存储。
  - 新用户默认状态为正常。
- 响应数据：
  - `userId`
  - `phone`
  - `nickname`
  - `status`

### 4.2 用户登录

- 请求方式：POST
- 请求路径：`/api/user/auth/login`
- 权限：无需登录
- 请求参数：
  - `phone`：手机号
  - `password`：密码
- 业务规则：
  - 手机号和密码必须匹配。
  - 用户状态必须为正常。
  - 登录成功后生成 Token。
- 响应数据：
  - `userId`
  - `phone`
  - `nickname`
  - `status`
  - `token`

### 4.3 员工登录

- 请求方式：POST
- 请求路径：`/api/admin/auth/login`
- 权限：无需登录
- 请求参数：
  - `phone`：手机号
  - `password`：密码
- 业务规则：
  - 手机号和密码必须匹配。
  - 员工状态必须为正常。
  - 登录成功后生成 Token，并在 Token 中区分店员或店长角色。
- 响应数据：
  - `employeeId`
  - `phone`
  - `name`
  - `role`
  - `status`
  - `token`

### 4.4 用户查看桌位列表

- 请求方式：GET
- 请求路径：`/api/user/tables`
- 权限：普通用户
- 请求参数：无
- 业务规则：
  - 用户必须登录。
  - 返回当前桌位状态。
  - 用户端可以展示 `sessionId`，但只有当前用户自己的开台会话才允许后续操作。
- 响应数据：
  - `tableList`
    - `tableId`
    - `tableNo`
    - `capacity`
    - `locationDesc`
    - `status`
    - `currentSessionId`

### 4.5 创建预约

- 请求方式：POST
- 请求路径：`/api/user/reservations`
- 权限：普通用户
- 请求参数：
  - `tableId`：桌位 ID
  - `peopleCount`：预约人数
  - `reserveTime`：预约到店时间
- 业务规则：
  - 用户必须登录。
  - 桌位必须为空闲。
  - 禁用桌位不允许预约。
  - 预约人数不能超过桌位容量。
  - 预约成功后，预约状态为待到店。
  - 预约成功后，桌位状态变为已预约。
- 响应数据：
  - `reservationId`
  - `reservationNo`
  - `tableId`
  - `tableNo`
  - `peopleCount`
  - `reserveTime`
  - `status`

### 4.6 查看我的预约列表

- 请求方式：GET
- 请求路径：`/api/user/reservations`
- 权限：普通用户
- 请求参数：
  - `status`：预约状态，可选
- 业务规则：
  - 用户必须登录。
  - 只能查看当前登录用户自己的预约。
- 响应数据：
  - `reservationList`
    - `reservationId`
    - `reservationNo`
    - `tableId`
    - `tableNo`
    - `peopleCount`
    - `reserveTime`
    - `status`

### 4.7 取消我的预约

- 请求方式：POST
- 请求路径：`/api/user/reservations/{reservationId}/cancel`
- 权限：普通用户
- 请求参数：
  - `reservationId`：预约 ID，路径参数
- 业务规则：
  - 用户必须登录。
  - 只能取消自己的预约。
  - 只能取消待到店状态的预约。
  - 取消成功后，预约状态变为已取消。
  - 取消成功后，桌位状态恢复为空闲。
- 响应数据：
  - `reservationId`
  - `reservationNo`
  - `status`
  - `tableId`
  - `tableStatus`

### 4.8 预约用户扫码到店

- 请求方式：POST
- 请求路径：`/api/user/reservations/{reservationId}/check-in`
- 权限：普通用户
- 请求参数：
  - `reservationId`：预约 ID，路径参数
  - `tableId`：扫码得到的桌位 ID
- 业务规则：
  - 用户必须登录。
  - 预约必须属于当前登录用户。
  - 预约状态必须为待到店。
  - 扫码桌位必须与预约桌位一致。
  - 核验成功后，预约状态变为已到店。
  - 创建开台会话，开台状态为等待中。
  - 桌位状态变为等待中。
- 响应数据：
  - `sessionId`
  - `sessionNo`
  - `reservationId`
  - `tableId`
  - `tableNo`
  - `sessionStatus`
  - `tableStatus`

### 4.9 未预约用户扫码占座

- 请求方式：POST
- 请求路径：`/api/user/tables/{tableId}/sessions`
- 权限：普通用户
- 请求参数：
  - `tableId`：扫码得到的桌位 ID，路径参数
- 业务规则：
  - 用户必须登录。
  - 桌位必须为空闲。
  - 禁用桌位不允许扫码占座。
  - 创建开台会话，开台状态为等待中。
  - 桌位状态变为等待中。
- 响应数据：
  - `sessionId`
  - `sessionNo`
  - `tableId`
  - `tableNo`
  - `sessionStatus`
  - `tableStatus`

### 4.10 查看启售菜品列表

- 请求方式：GET
- 请求路径：`/api/user/dishes`
- 权限：普通用户
- 请求参数：
  - `categoryId`：菜品分类 ID，可选
- 业务规则：
  - 用户必须登录。
  - 只返回启售状态的菜品。
- 响应数据：
  - `dishList`
    - `dishId`
    - `categoryId`
    - `name`
    - `price`
    - `image`
    - `description`
    - `status`

### 4.11 创建堂食订单

- 请求方式：POST
- 请求路径：`/api/user/sessions/{sessionId}/orders`
- 权限：普通用户
- 请求参数：
  - `sessionId`：开台会话 ID，路径参数
  - `items`：订单菜品列表
    - `dishId`
    - `quantity`
    - `remark`：口味或备注，可选
- 业务规则：
  - 用户必须登录。
  - 开台会话必须存在。
  - 开台会话必须属于当前登录用户。
  - 开台会话状态必须为等待中。
  - 桌位状态必须为等待中。
  - 菜品必须存在且处于启售状态。
  - 菜品数量必须大于 0。
  - 创建订单和订单明细。
  - 订单状态为已下单。
  - 开台会话状态变为用餐中。
  - 桌位状态变为用餐中。
- 响应数据：
  - `orderId`
  - `orderNo`
  - `sessionId`
  - `tableId`
  - `tableNo`
  - `totalAmount`
  - `status`
  - `items`
    - `dishId`
    - `dishName`
    - `dishPrice`
    - `quantity`
    - `amount`

### 4.15 商户更新订单状态

- 请求方式：POST
- 请求路径：`/api/admin/orders/{orderId}/status`
- 权限：店员、店长
- 请求参数：
  - `orderId`：订单 ID，路径参数
  - `status`：目标订单状态
- 业务规则：
  - 员工必须登录。
  - 订单必须存在。
  - V1 允许按正常顺序更新：已下单 -> 制作中 -> 已上齐 -> 已完成。
  - 已取消订单不允许继续更新。
  - 已完成订单不允许继续更新。
- 响应数据：
  - `orderId`
  - `orderNo`
  - `status`

### 4.16 店员清台释放桌位

- 请求方式：POST
- 请求路径：`/api/admin/sessions/{sessionId}/close`
- 权限：店员、店长
- 请求参数：
  - `sessionId`：开台会话 ID，路径参数
- 业务规则：
  - 员工必须登录。
  - 开台会话必须存在。
  - 开台状态必须为用餐中。
  - 若有关联订单，订单状态应更新为已完成。
  - 开台状态更新为已完成。
  - 桌位状态恢复为空闲。
  - 桌位当前开台会话 ID 清空。
- 响应数据：
  - `sessionId`
  - `sessionNo`
  - `tableId`
  - `tableNo`
  - `sessionStatus`
  - `tableStatus`

### 4.17 店员释放未下单的等待中桌位

- 请求方式：POST
- 请求路径：`/api/admin/sessions/{sessionId}/cancel`
- 权限：店员、店长
- 请求参数：
  - `sessionId`：开台会话 ID，路径参数
- 业务规则：
  - 员工必须登录。
  - 开台会话必须存在。
  - 开台状态必须为等待中。
  - 店员应先线下确认用户已经离开。
  - 开台状态更新为已取消。
  - 桌位状态恢复为空闲。
  - 桌位当前开台会话 ID 清空。
- 响应数据：
  - `sessionId`
  - `sessionNo`
  - `tableId`
  - `tableNo`
  - `sessionStatus`
  - `tableStatus`

## 5. 商户端基础管理接口清单

### 5.1 桌位管理

- `GET /api/admin/tables`：查看桌位列表，店员、店长
- `GET /api/admin/tables/{tableId}`：查看桌位详情，店员、店长
- `POST /api/admin/tables`：新增桌位，店员、店长
- `PUT /api/admin/tables/{tableId}`：修改桌位，店员、店长
- `POST /api/admin/tables/{tableId}/disable`：禁用桌位，店员、店长
- `POST /api/admin/tables/{tableId}/enable`：启用桌位，店员、店长
- `DELETE /api/admin/tables/{tableId}`：删除桌位，店长

说明：

- 新增桌位默认状态为空闲。
- 禁用桌位要求当前桌位为空闲。
- 启用桌位要求当前桌位为禁用，启用后恢复为空闲。
- 删除桌位要求当前桌位为空闲或禁用；V1 建议优先使用禁用，谨慎物理删除。

### 5.2 预约管理

- `GET /api/admin/reservations`：查看预约列表，店员、店长
- `GET /api/admin/reservations/{reservationId}`：查看预约详情，店员、店长
- `POST /api/admin/reservations/{reservationId}/cancel`：取消异常预约，店员、店长

### 5.3 菜品管理

- `GET /api/admin/dishes`：查看菜品列表，店员、店长
- `GET /api/admin/dishes/{dishId}`：查看菜品详情，店员、店长
- `POST /api/admin/dishes`：新增菜品，店员、店长
- `PUT /api/admin/dishes/{dishId}`：修改菜品，店员、店长
- `POST /api/admin/dishes/{dishId}/status`：修改菜品状态，店员、店长
- `DELETE /api/admin/dishes/{dishId}`：删除菜品，店长

### 5.4 员工管理

- `GET /api/admin/employees`：查看员工列表，店长
- `GET /api/admin/employees/{employeeId}`：查看员工详情，店长
- `POST /api/admin/employees`：新增员工，店长
- `POST /api/admin/employees/{employeeId}/disable`：禁用员工，店长
- `POST /api/admin/employees/{employeeId}/enable`：启用员工，店长

## 6. 当前暂不设计的接口

- 短信验证码发送与登录接口。
- 支付接口。
- 购物车接口。
- 催单接口。
- 改单和退单接口。
- csv 导入导出接口。
- 排班接口。
- 操作日志查询接口。

这些接口放入 V2 或后续版本，避免 V1 范围失控。
