# V2 阶段开发计划

## 1. 当前状态

V1 主业务闭环已经完成，当前项目可以作为 V2 增强阶段的基础版本。

V1 已完成主链路：

```text
用户预约桌位
  -> 到店扫码开台
  -> 查看菜品
  -> 创建堂食订单
  -> 商户处理订单
  -> 店员清台释放桌位
```

V2 不推翻 V1 业务模型，而是在已有闭环上补强后端工程能力，重点围绕：

- 并发控制与数据一致性。
- 管理端分页查询。
- Redis 缓存真实落地。
- 自动化测试补强。
- Docker Compose 部署增强。
- README、过程资产、简历表达同步更新。

## 2. V2 阶段目标

V2 的目标不是继续堆业务功能，而是让项目从“能跑通主流程”升级为“具备工程可靠性和简历表达价值”。

阶段验收标准：

- V1 主流程仍稳定可运行。
- 至少一个关键并发问题被修复并可验证。
- 管理端主要列表接口支持分页。
- Redis 在真实业务场景中落地。
- Docker Compose 能编排 MySQL、Redis、后端服务。
- 至少补充核心业务测试、并发测试或缓存测试。
- README 能体现 V2 技术亮点。
- 过程资产能沉淀并发、Redis、分页、测试、部署相关经验。

推荐开发顺序：

```text
V2 基线检查
  -> 并发控制与数据一致性
  -> 管理端分页查询
  -> Redis 菜品缓存
  -> 自动化测试补强
  -> Docker Compose 部署增强
  -> 文档与复盘收束
```

## 3. 开发任务拆分

### 3.1 V2 基线检查

目标：确认 V1 当前状态可以作为 V2 起点。

需要完成：

- 创建 V2 开发分支。
- 执行构建：

```powershell
.\mvnw.cmd clean test
```

- 使用 Apifox 跑通 V1 主流程。
- 检查 Dockerfile、docker-compose.yml 是否仍可用。
- 明确 V2 不改动 V1 已稳定的核心业务模型。

验收标准：

- 构建通过。
- V1 主链路可运行。
- V2 分支建立。
- 当前遗留问题已记录到 V2 待办。

### 3.2 并发控制与数据一致性

目标：解决 V1 中关键状态流转在并发场景下不可靠的问题。

优先场景：

- 创建预约：避免同一桌位被重复预约。
- 预约扫码开台：避免重复核验、重复创建开台会话。
- 未预约扫码占座：避免多个用户同时占用同一桌位。
- 创建订单：避免同一等待中会话被重复下单。
- 清台释放桌位：避免重复清台或状态覆盖。

建议实现方向：

- 优先使用数据库条件更新：

```sql
update dining_table
set status = ?
where id = ? and status = ?
```

- 根据更新行数判断并发冲突。
- 对关键表评估乐观锁字段 `version`。
- 对开台会话补充活跃状态约束，例如 `active_flag`。
- 优先掌握事务、条件更新、乐观锁，不急于使用 Redis 分布式锁。

验收标准：

- 至少一个并发场景被修复并验证。
- 至少两个核心状态流转改为更可靠的条件更新。
- V1 主流程不被破坏。

### 3.3 管理端分页查询

目标：补齐管理端常见列表能力，训练 MyBatis-Plus 分页和动态查询。

优先改造接口：

- 管理端订单列表。
- 管理端预约列表。
- 管理端菜品列表。
- 管理端开台会话列表。
- 管理端员工列表。
- 管理端桌位列表。

统一约定：

- 请求参数包含：

```text
pageNo
pageSize
```

- 筛选条件继续使用 query 参数或 DTO 绑定。
- 响应统一使用：

```text
PageResult<T>
```

建议顺序：

1. 先以订单列表作为样板。
2. 确认 MyBatis-Plus 分页插件配置。
3. 固化分页 DTO 和 `PageResult` 使用方式。
4. 推广到预约、菜品、开台会话、员工、桌位。
5. 同步 Knife4j 注解和接口文档。

验收标准：

- 至少 4 个管理端列表接口完成分页。
- 分页参数和返回结构统一。
- Apifox / Knife4j 能清楚展示分页接口。

### 3.4 Redis 缓存落地

目标：在真实业务场景中引入 Redis，练习缓存读取、缓存失效和 Docker Compose 集成。

优先场景：

```text
用户端启售菜品列表缓存
```

选择原因：

- 查询频率高。
- 数据读多写少。
- 和当前业务自然相关。
- 菜品新增、修改、删除、上下架时有明确缓存失效点。

建议实现：

- 引入 Spring Data Redis。
- docker-compose.yml 增加 Redis 服务。
- 配置 Redis 连接。
- 用户端查询启售菜品列表：
  - 先查 Redis。
  - 命中直接返回。
  - 未命中查数据库并写缓存。
- 管理端菜品变更后删除相关缓存。

建议 key：

```text
dish:on-sale:all
dish:on-sale:category:{categoryId}
```

暂不做：

- 复杂缓存预热。
- 多级缓存。
- 分布式锁。
- 缓存雪崩、穿透、击穿完整治理。

验收标准：

- Redis 容器可启动。
- 启售菜品列表能命中缓存。
- 菜品变更后缓存能失效。
- 日志或测试能证明缓存生效。

### 3.5 自动化测试补强

目标：从主要依赖 Apifox 手动测试，升级到具备基础自动化验证能力。

优先测试范围：

- 核心业务流程集成测试。
- 并发场景测试。
- Redis 缓存测试。
- 权限拦截关键测试。

建议顺序：

1. 先补核心业务测试：
   - 创建预约。
   - 扫码开台。
   - 创建订单。
   - 清台。
2. 再补并发测试：
   - 并发预约同一桌位。
   - 并发创建订单。
3. 最后补缓存测试：
   - 首次查询写入缓存。
   - 二次查询命中缓存。
   - 菜品变更后缓存失效。

验收标准：

- 至少 3 个核心业务测试。
- 至少 1 个并发测试。
- 至少 1 个 Redis 缓存测试。
- `mvn clean test` 可稳定运行。

### 3.6 Docker Compose 部署增强

目标：让项目具备更完整的部署闭环。

当前已有：

- Dockerfile。
- docker-compose.yml。
- MySQL 容器。
- 后端服务容器。
- 服务器本地打包 jar 的部署流程。

V2 增强：

- docker-compose.yml 增加 Redis。
- README 增加 Redis 部署说明。
- 验证 Linux 环境部署流程：

```bash
git pull
chmod +x mvnw
./mvnw clean package -DskipTests
cp target/food-flow-manager-0.0.1-SNAPSHOT.jar food-flow-manager.jar
docker compose up -d --build
```

- 验证：
  - `docker compose ps`
  - 应用日志
  - MySQL 日志
  - Redis 容器状态
  - `/actuator/health`
  - Knife4j 页面

暂不强制：

- CI/CD。
- 镜像仓库。
- Nginx。
- HTTPS。
- K8s。

验收标准：

- MySQL、Redis、后端服务可通过 compose 启动。
- 后端服务可连接 MySQL 和 Redis。
- README 能指导自己重新部署一次。



- 复杂预约时间窗口。
- 微服务拆分。

## 6. 关键假设

- V2 阶段主攻后端能力，不做前端 MVP。
- Spring Security 不作为 V2 前半段强制任务。
- RabbitMQ 不作为 V2 核心必做项。
- Redis 优先落地启售菜品缓存，不优先做验证码或分布式锁。
- Docker 部署继续采用服务器本地打包 jar 的方式，不引入 CI/CD。
- V2 不推翻 V1 业务模型，只做增强和工程质量提升。
